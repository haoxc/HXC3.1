# MOC / Wiki 目录页规范

> [!summary]
> MOC / Wiki 索引页负责导航，不负责承载正文内容。它应该像地图，而不是像文章。

## 核心规则

MOC / Wiki 目录页只保留：

- 层级目录；
- Wiki 链接；
- 极少量分类词；
- 必要时保留简短别名或入口说明。

MOC / Wiki 目录页不放：

- 大段解释；
- 操作步骤；
- 配置细节；
- 长表格；
- 经验总结正文；
- 完整备忘录。

## 推荐形态

```markdown
- 理念
	- 诚实(Honest)
	- 无害(Harmless)
	- 有用(Helpful)
- 亮点
	- Artifacts
	- 长上下文(Long Context window)
	- [[思维画布(Thinking Canvas)]]
- 学习规划
	- [[学习实战]]
	- [[claude-学习路径]]
- 实践
	- 准备
		- [[claude-安装]]
	- 指南
		- [[claude-code-使用备忘录]]
	- 工具
		- [[claude工具-切换模型-cc-switch]]
```

## 内容拆分规则

当需要写具体内容时，拆到独立笔记：

| 内容类型 | 建议落点 |
| --- | --- |
| 使用备忘录 | `xxx-使用备忘录.md` |
| 配置说明 | `xxx-配置.md` |
| 操作流程 | `xxx-操作流程.md` |
| 概念定义 | `xxx.md` 或术语库 |
| 工具对比 | `xxx-对比.md` |
| 问题排查 | `xxx-问题排查.md` |

MOC 页只链接这些笔记，不展开正文。

## 判断标准

如果一个 MOC / Wiki 页出现以下情况，就应拆分：

1. 单个节点下面出现超过 3 行解释。
2. 页面出现完整配置代码块。
3. 页面出现长表格。
4. 页面需要滚动很久才能看完目录。
5. 某段内容可以独立复用、引用或维护。

## 修改原则

编辑 MOC / Wiki 页时：

- 优先维护结构，而不是补正文；
- 新增内容先判断是否应该新建独立笔记；
- 链接命名要清晰，让读者只看目录就知道去哪里；
- 不为“说明清楚”而牺牲目录页的轻量性。

## 压缩记忆

> MOC / Wiki 页是地图，不是正文。地图只放路标；内容放到独立笔记。
