# 当前 Vault 审计报告-2026-05-01

> [!summary]
> 当前 Vault 顶层结构基本符合 HXC Vault 3.1，主要风险不在目录缺失，而在 MOC 断链、Frontmatter 质量不一致、`type` 枚举扩散、`---/` 工作区命名不稳定。大附件主要位于 `98-附件/` 且已被 `.gitignore` 排除，暂未形成 Git 推送阻塞类 P1。

## 审计口径

- 主规则源：Hermes skill `pjx-common-auditor`
- 本地规则页：[[项目审计规则]]
- 当前类型判定：知识管理型 Vault
- 扫描范围：Vault 根目录下非隐藏知识库内容；隐藏工具目录仅作为环境风险观察，不作为知识目录违规主证据
- 扫描规模：约 2615 个 Markdown 文件

## 结论

当前 Vault 的一级目录体系已经成型，`01-我的/`、`02-输入/`、`03-领域/`、`04-笔记/`、`05-资源/`、`06-商业/`、`09-工具/`、`10-归档/`、`98-附件/`、`99-设置/`、`---/`、`z附件/` 均存在。后续治理重点应从“补目录”转向“清链接、稳元数据、统一 MOC 与工作区命名”。

## Findings

### [P1] 核心 MOC 存在断链

Evidence:

- MOC 文件约 30 个。
- MOC Wikilink 断链约 6 条。
- 示例：
  - `01-我的/01-我的.md` → `[[日常笔记]]`
  - `01-我的/学习/学习.md` → `[[语言(Language)]]`
  - `01-我的/学习/表达/表达.md` → `[[表达-现象描述]]`
  - `01-我的/学术/学术.md` → `[[26-Paper Review]]`
  - `01-我的/课题/课题.md` → `[[科学计算]]`

Why it matters:

核心 MOC 是知识库入口。Hermes 规则将 MOC 断链归为 P1，因为它会直接破坏导航和发现路径。

Fix:

优先修复核心 MOC 断链：要么创建目标笔记，要么改链到现有笔记，要么删除已经失效的入口。

### [P2] 根目录存在非标准入口 `Clippings`

Evidence:

- 标准顶层目录均存在。
- 非隐藏、非标准顶层目录：`Clippings`

Why it matters:

Hermes KV-1 将未定义顶层目录视为 P2。`Clippings` 可能是浏览器剪藏入口，但它与 `02-输入/` 的信息摄入职责重叠，长期会形成双入口。

Fix:

确认 `Clippings` 的角色：

- 若是临时收件箱，迁移到 `02-输入/` 下的剪藏或草稿目录；
- 若是工具自动同步目录，保留但在规则页中声明为工具例外；
- 若已废弃，归档到 `10-归档/` 或清理。

### [P2] Frontmatter 完整性不稳定

Evidence:

- 无 Frontmatter 的 Markdown 文件约 169 个。
- Frontmatter 存在但缺字段的文件约 70 个。
- 常见缺失字段：`create-date`、`description`。
- 示例：
  - `---/术语库(Terms svy)/01-一般术语知识库(Common DB)/KANO 模型.md`
  - `---/TEST/TEST-知识提示词/练习-知识卡片.md`
  - `---/术语库(Terms svy)/01-一般术语知识库(Common DB)/维度(Dimension).md`

Why it matters:

Frontmatter 是跨目录检索、筛选和自动审计的基础。缺字段会降低可检索性，也会让后续批量整理难以判断笔记类型和创建时间。

Fix:

分批补齐：

1. 先处理新近活跃目录和核心 MOC。
2. 再处理 `---/术语库(Terms svy)/`、`01-我的/`、`03-领域/` 的核心笔记。
3. 对 `._` 开头的 macOS 残留文件单独判断是否需要清理。

### [P2] `type` 字段枚举扩散

Evidence:

- 空 `type` 约 933 个。
- 非 Hermes 基础枚举的 `type` 约 293 个。
- `type: MOC` 大写约 21 个；Hermes 主规则要求 `type: moc`。
- 常见扩展值包括：`term`、`日记(Diary)`、`概念定义`、`course`、`调研报告`、`规约`。

Why it matters:

Hermes KV-4 基础枚举为 `note/moc/log/reference/template`。当前扩展值可表达更细语义，但若没有映射表，会导致自动审计、筛选和跨工具解析不稳定。

Fix:

建立 `type` 映射表：

| 当前类型 | 建议基础类型 | 备注 |
| --- | --- | --- |
| `MOC` | `moc` | 统一小写 |
| `term`、`概念定义` | `note` | 可用 tags 表达术语类型 |
| `日记(Diary)` | `log` | 周期记录归 log |
| `调研报告`、`调研摘要` | `note` | 可用 tags 或 `report_type` 扩展 |
| `规约` | `note` 或 `reference` | 若为本 Vault 有效规则，建议另设 `standard: true` |

### [P2] `---/` 工作区命名不稳定

Evidence:

`---/` 下当前一级工作区包括：

- `-workspaces`
- `.space`
- `2026-04-30-Vault整改`
- `TEST`
- `主题-世界模型场景`
- `术语库(Terms svy)`

其中非日期前缀且非系统目录的工作区：

- `TEST`
- `主题-世界模型场景`
- `术语库(Terms svy)`

Why it matters:

Hermes KV-6/KV-8 要求项目目录在需要按时间检索时使用日期前缀。无日期工作区会削弱项目生命周期检索，也会让临时区和长期库混在一起。

Fix:

- `TEST`：若无保留价值，迁移到归档或删除前先确认。
- `主题-世界模型场景`：若是长期主题，迁移到 `03-领域/`；若是项目，补日期前缀和入口索引。
- `术语库(Terms svy)`：若是长期术语库，应考虑迁移到 `---/术语库(Terms svy)` 的例外规则或正式化到 `03-领域/` / `05-资源/` / `99-设置/` 中的固定位置。

### [P3] 长文件数量较多，需要区分正文过载与生成资产

Evidence:

- 超过 80 行的 Markdown 文件约 317 个。
- 样本中包含会议黑板、UML 导出、Excalidraw、长需求分析等。

Why it matters:

Hermes KV-2 将超过 80 行且无拆分的普通正文笔记视为 P2；但 Excalidraw、导出文件、会议原始黑板不应直接套用普通笔记规则。

Fix:

先分类后处理：

- `*.excalidraw.md`、UML 导出：视为资源/生成资产，不按普通笔记长度直接处罚。
- 会议黑板和长需求分析：补入口摘要、结论段和子页面链接。
- 普通知识笔记：超过 80 行则拆分或增加 `## 细节` 收束。

## 非问题项

### `04-笔记/` 当前未堆积

扫描结果：`04-笔记/` 下 Markdown 数量为 0。Hermes KV-8 中“收件箱堆积”问题暂不成立。

### 大附件 Git 风险暂未触发

扫描发现大量超过 10MB 的附件位于 `98-附件/`。当前 Git 只跟踪 `98-附件/98-附件.md`，且 `.gitignore` 已忽略 `98-附件/*`。因此暂未形成“≥10MB 文件进入 Git 跟踪路径”的 P1。

## 建议调整

| 优先级 | 当前 | 建议 | 原因 |
| --- | --- | --- | --- |
| P1 | 核心 MOC 断链 | 逐条修复 MOC 链接 | 恢复导航入口 |
| P2 | `Clippings/` | 确认为工具例外或迁入 `02-输入/` | 避免双入口 |
| P2 | 空 `type` 与扩展 `type` | 建立映射表后批量规范 | 提升自动审计稳定性 |
| P2 | `---/TEST` 等无日期工作区 | 确认保留/归档/迁移策略 | 降低临时区噪音 |
| P3 | 长文件 | 区分生成资产与正文笔记 | 避免误伤导出文件 |
