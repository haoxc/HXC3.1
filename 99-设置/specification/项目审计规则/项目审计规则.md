# 项目审计规则

> [!summary]
> 当前 Vault 的审计以 Hermes skill `pjx-common-auditor` 为主规则源。发生冲突时，Hermes 规则优先；本地规范用于补充细节，不能覆盖 Hermes 已明确的判定。

## 适用对象

当前工作区具备以下特征：

- 存在 `01-我的/`、`02-输入/`、`03-领域/`
- 存在 `09-工具/`、`99-设置/`、`---/`
- 顶层结构更接近 Obsidian 知识管理 Vault，而不是单一软件研发项目

因此默认审计模型为：**知识管理型 Vault 审计**。

软件研发/PJX 项目规则仅在以下范围启用：

- 子项目明确存在 `00-总览/`、`10-需求/`、`20-模块/` 等研发项目结构；
- 用户明确要求按 Agent / Skill / Prompt / 软件研发项目审计；
- 某个 `---/` 工作区或项目目录具备交付、需求、模块、治理、发布等生命周期对象。

## 规则优先级

审计时按以下顺序裁决：

1. Hermes skill `pjx-common-auditor` 明确定义的规则。
2. 当前 Vault 的 `CLAUDE.md`、`99-设置/specification/` 中的有效规范。
3. 当前对话中用户确认的补充规则。

当本地旧规范与 Hermes 规则冲突时：

- 审计结论以 Hermes 为准；
- 本地旧规范作为历史参考或项目特例；
- 不直接移动、重命名或删除文件，除非用户明确要求实施。

当规则没有被以上来源覆盖时：

- 不强行分级为 P1/P2/P3；
- 先说明“规则未定义”；
- 列出可选方案；
- 等用户确认后再纳入审计。

## 严重级别

| 级别 | 判定含义 | 典型影响 |
| --- | --- | --- |
| P1 | 结构、导航、引用或推送会被严重破坏 | 核心 MOC 断链、大文件阻塞 Git、入口不存在 |
| P2 | 检索、迁移、审查、维护质量明显下降 | 目录角色错位、Frontmatter 缺失、项目无入口 |
| P3 | 低风险不一致或可选优化 | 标签轻微不统一、命名风格不够一致 |

## 审计流程

1. 识别目标类型：知识 Vault、软件研发项目、混合结构或未定义结构。
2. 构建目录映射：只抽样必要目录，不预设全部违规。
3. 对照对应规则：知识 Vault 使用 KV 规则；软件项目使用 PJX 规则。
4. 输出发现：每条发现必须有证据、影响、建议修复。
5. 区分立即修复与可选清理。
6. 修复后复查 P1；P2/P3 按用户选择处理。

## 知识 Vault 审计规则

### KV-1 顶层结构

当前 Vault 采用 HXC Vault 3.1 结构，标准顶层目录为：

| 目录 | 角色 | 缺失默认级别 |
| --- | --- | --- |
| `01-我的/` | 个人信息、目标、日记、思考 | P2；若个人周期记录也缺失则 P1 |
| `02-输入/` | 阅读摘录、外部文章、信息摄入 | P2 |
| `03-领域/` | 知识领域与主题笔记 | P2 |
| `04-笔记/` | 零散笔记、待整理收件箱 | P2 |
| `05-资源/` | 资源清单、工具清单、书单 | P2 |
| `06-商业/` | 公司、产品、市场、SaaS 等商业情报 | P3 |
| `09-工具/` | 具体工具使用笔记 | P2 |
| `10-归档/` | 过时或不再维护的笔记 | P2 |
| `98-附件/` | 图片、PDF、音视频等静态资源 | P2 |
| `99-设置/` | 配置、模板、规范、系统设置 | P2 |
| `---/` | 工作区、项目、临时交付空间 | P3 |
| `z附件/` | 额外附件暂存 | P3 |

根目录出现未定义业务目录时，默认按 P2 处理；工具操作笔记进入 `09-工具/`，领域原理进入 `03-领域/`，项目产出进入 `---/` 下的项目/工作区。

### KV-2 笔记质量

笔记遵循 `CLAUDE.md` 的写作规则：

- 标题见名知意，避免“关于”“如何”等弱信息词。
- 核心结论应在前 15 行内出现。
- 超过 80 行应考虑拆分子页面。
- 结论需要理由，技术决策类笔记建议使用“结论、理由、出处”三段式。
- 禁止口语化、情绪化、宣传式措辞。
- 写完事实即停止，不额外添加空泛总结段。

默认分级：

| 问题 | 级别 |
| --- | --- |
| 核心结论不在前 15 行 | P2 |
| 无理由的断言式建议 | P2 |
| 超过 80 行且无拆分/子页面 | P2 |
| 口语化或情绪化词语 | P3 |

### KV-3 MOC 信息负载

MOC 是导航表面，不是正文、日志、决策记录或管理台账。

MOC 必须满足：

- Hermes 主规则要求命名为 `_MOC_领域名.md`；
- Frontmatter 中 `type` 使用 `moc`；
- 至少有一个 `##` 分组或等价的目录结构；
- 只放当前层级入口、直接上游、子主题链接和极简说明；
- 不放长篇正文、进度日志、决策记录、完整规范条款；
- 不链接大量下游深层文件。

当前 Vault 中的 folder-note MOC 属于本地历史规范。审计时应标注为“与 Hermes 命名口径冲突”，不自动重命名；是否迁移为 `_MOC_` 体系需由用户单独确认。

默认分级：

| 问题 | 级别 |
| --- | --- |
| 核心 MOC 链接断裂 | P1 |
| MOC 正文超过 50 行 | P1 |
| MOC 正文超过 20 行 | P2 |
| MOC 链接数超过 40 且未拆分 | P2 |
| MOC 中放进度、决策、日志 | P2 |
| 无分组 MOC | P3 |

### KV-4 Frontmatter

普通笔记建议具备以下字段：

```yaml
---
tags: [分类标签]
description: 一句话说明核心内容
type: note
create-date: YYYY-MM-DD
---
```

允许的 `type` 基础枚举为：

- `note`
- `moc`
- `log`
- `reference`
- `template`

默认分级：

| 问题 | 级别 |
| --- | --- |
| 新笔记无 Frontmatter | P1 |
| 存量笔记无 Frontmatter | P2 |
| 缺少 `type` 或 `create-date` | P2 |
| `type` 不在枚举内或与用途不匹配 | P2 |
| 缺少 `tags` 或 `description` | P3 |

### KV-5 链接质量

内部笔记默认使用 Obsidian Wikilink：

- 首次出现时链接，同篇不重复滥链；
- 链接文本优先等于目标笔记标题；
- 别名仅用于消歧义或语法需要；
- 附件优先用 Wikilink 引用；
- 核心 MOC、核心入口笔记不得断链。

默认分级：

| 问题 | 级别 |
| --- | --- |
| 核心笔记或核心 MOC 断链 | P1 |
| 普通笔记断链 | P2 |
| 附件裸路径导致 Obsidian 无法跟踪 | P2 |
| 同篇重复链接超过 3 次 | P3 |
| 无意义别名 | P3 |

### KV-6 命名规则

知识 Vault 命名以“可检索、可读、可长期维护”为准：

- 文件名应等于或接近笔记标题；
- 中文为主，英文术语可用括号补充；
- 避免空格和特殊字符；
- 日记使用 `日记-YYYY-MM-DD.md`；
- 周报使用 `周报-YYYY-Www.md`；
- 月报使用 `月报-YYYY-MM.md`；
- 项目目录在需要按时间检索时使用日期前缀；
- 附件目录统一使用 `z附件` 或根级 `98-附件/`。

默认分级：

| 问题 | 级别 |
| --- | --- |
| MOC 命名或类型无法识别 | P2 |
| 项目目录缺少必要日期前缀 | P2 |
| 日记/周期笔记命名不符合格式 | P2 |
| 附件目录出现 `附件`、`assets`、`zAttachments` 等变体 | P2 |
| 文件名含空格或特殊字符 | P3 |

### KV-7 标签一致性

标签用于检索，不用于情绪表达或随手分类：

- 领域标签保持一致，如 `AI-Agent`、`编程-Python`；
- 类型标签使用统一中文词，如 `教程`、`速查`、`对比`；
- 状态标签使用统一词，如 `草稿`、`已审阅`、`过时`；
- 人员标签使用 `人员/xxx`；
- 同一概念不得长期存在多个大小写或中英文变体。

默认分级：

| 问题 | 级别 |
| --- | --- |
| 同一概念存在多个标签变体 | P2 |
| 标签中英文混用但无规律 | P3 |
| 使用非推荐标签但未形成替代体系 | P3 |

### KV-8 工作区卫生

收件箱、归档区、项目工作区应保持角色清晰：

- `04-笔记/` 是临时收件箱，不应长期堆积；
- `10-归档/` 条目应有归档原因或日期；
- `---/` 下活跃项目应有入口索引；
- 已完成且不再维护的工作区应迁移或标注状态。

默认分级：

| 问题 | 级别 |
| --- | --- |
| `04-笔记/` 堆积超过 50 篇且无近期整理痕迹 | P2 |
| `---/` 下无日期前缀项目目录超过 3 个 | P2 |
| 过时内容超过 1 年仍无归档状态 | P3 |

### KV-9 交叉引用完整性

知识库中的文件需要至少有一条发现路径：

- MOC 引用；
- 反向链接；
- 有意义的 tags；
- 项目入口索引；
- SVG 等图形资产应有同名或配套说明页。

默认分级：

| 问题 | 级别 |
| --- | --- |
| MOC 中链接断裂 | P1 |
| 普通笔记断链 | P2 |
| SVG 缺少配套说明页 | P2 |
| 笔记无 MOC、无反链、无有效 tags | P2；大 Vault 中可升 P1 |
| `---/` 项目无入口索引 | P3 |

### KV-10 附件管理

附件规则以 Git 可维护性和 Obsidian 可追踪性为准：

- 普通局部附件进入同级 `z附件/`；
- 大文件、PDF、音视频进入根级 `98-附件/`；
- 大文件不应进入 Git 跟踪路径；
- 移动附件后应保留 Wikilink 或说明页，避免断链。

默认分级：

| 问题 | 级别 |
| --- | --- |
| ≥10MB 文件留在 Git 跟踪路径且不在 `98-附件/` | P1 |
| 附件目录使用非标准变体 | P2 |
| 移动附件后无占位链接或说明 | P2 |

### KV-11 笔记效用

笔记应服务于理解、决策、创作或执行，而不是被动囤积信息：

- `description` 应说明用途或回答的问题；
- 教程/操作型笔记应有可执行步骤、预期结果、常见错误；
- 对比/选型型笔记应有明确结论；
- `02-输入/` 摘录应有个人批注或指向产出笔记；
- 成型知识笔记应能独立理解，必要前置信息通过链接补齐。

默认分级：

| 问题 | 级别 |
| --- | --- |
| `02-输入/` 超过 20 篇且零 outgoing link | P2 |
| 教程/操作笔记无可执行步骤 | P2 |
| `04-笔记/` 超过 100 篇且 3 个月无迁移 | P2 |
| `02-输入/` 纯粘贴无加工超过 10 篇 | P3 |
| 选型笔记无结论 | P3 |

## 软件研发/PJX 子项目审计规则

当目标目录被判定为软件研发项目时，使用 Hermes `pjx-common-auditor/references/naming-rules.md` 作为主规则源。

### 标准结构

基线项目：

```text
00-总览
01-规范
05-协同
09-治理
10-管理
20-模块
60-资产
90-归档
98-附件
99-设置
```

需求密集型项目：

```text
00-总览
01-规范
05-协同
09-治理
10-需求
15-管理
20-模块
60-资产
90-归档
98-附件
99-设置
```

### 关键规则

| 规则 | Hermes 判定 |
| --- | --- |
| `00-总览` 是当前唯一标准入口 | 新项目不得使用 `00-总控`；历史项目不强制重命名 |
| `02-治理` 是遗留/兼容结构 | 新 ZS/PJX 项目默认使用 `09-治理` |
| `01-规范` 与 `09-治理` 不合并 | 规范放有效规则，治理放证据、ADR、变更记录 |
| ADR 源记录放 `09-治理/` | 管理区只能做索引，不保存 ADR 正文 |
| 顶层 `10-需求` 存在时，模块内优先用 `承接/` | 避免业务需求与模块技术交接混淆 |
| 第三层及更深目录默认不用数字前缀 | 使用 `方案`、`参考`、`设计`、`开发` 等语义名 |
| 计划类稳定文件用版本号 | 如 `PLAN-V001-{topic}.md` |
| 一次性证据、检查、验证记录用日期 | 如 `{topic}-{yy.MMdd}.md` |

## 与现有规范的冲突处理

| 旧/本地规则 | Hermes 主规则 | 当前处理 |
| --- | --- | --- |
| `项目目录规范` 中的 PARA 风格 `00_项目主控台` | 软件研发/PJX 新项目使用 `00-总览` | PARA 规则保留为历史/非 PJX 项目参考 |
| 早期项目使用 `00-总控` | `00-总控` 已弃用，新项目使用 `00-总览` | 历史项目不自动重命名 |
| 早期治理目录使用 `02-治理` | 新项目默认使用 `09-治理` | 仅作为遗留兼容，不推荐新建 |
| folder-note MOC 与 `_MOC_` 命名并存 | Hermes 要求 `_MOC_领域名.md` 且 `type: moc` | 作为命名冲突记录；不自动批量改名，迁移需用户确认 |
| 本地 `type: 规约` 等扩展类型 | Hermes 基础枚举为 `note/moc/log/reference/template` | 作为类型扩展冲突记录；普通知识笔记按 Hermes 枚举审计 |

## 禁止行为

- 不把软件研发项目规则套用到所有知识笔记目录。
- 不把 MOC 当作正文页、日志页、决策页或台账页。
- 不把当前偏好当作已确认规则。
- 不在规则未定义时移动、重命名或删除文件。
- 不把历史 ADR 或历史项目命名直接判为缺陷，除非影响当前导航或当前规则。
- 不把附件、大文件、客户原始材料直接推入 Git 跟踪路径。

## 输出模板

审计输出应使用以下结构：

```markdown
**结论**
一句话说明整体结构健康度和主要风险。

**Findings**
- [P1/P2/P3] 问题标题
  - Evidence: 路径、行号或目录证据
  - Why it matters: 具体影响
  - Fix: 精确修复建议

**建议调整**
| 当前 | 建议 | 原因 |
| --- | --- | --- |

**迁移适配**
- 可复用内容
- 项目特例
- 应保留为历史的内容

**验证命令**
- 复查命令或检查方式
```
