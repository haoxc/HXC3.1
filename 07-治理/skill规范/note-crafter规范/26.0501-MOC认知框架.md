# 26.0501-MOC认知框架

## 背景

note-crafter skill 的 MOC 页面规范此前只有机械约束（禁止长文、type=moc、至少一个 ## 分组），缺少**分组逻辑的设计原则**。用户提出两个维度：

1. Wiki 形式连接知识点
2. 建立认知框架分类并排序内容

## 落地方案

### 声明层：vault-rules.yaml `moc` 块追加

新增两个子块：

`moc.design_principles` — 三条设计原则：

| 原则 | 含义 |
|------|------|
| `wiki_hub` | MOC = Wiki hub page，每个 ## 分组是概念聚类，链接间通过分组隐含语义 |
| `cognitive_frame` | 分组反映认知模型，不从文件系统平铺 |
| `navigation_only` | MOC 仍是导航页，认知框架通过分组结构表达，不靠解释性段落 |

`moc.grouping_strategies` — 四种分组策略枚举，每种含 `description` / `applies` / `example` / `anti_example`：

| 策略 | 适用 | 示例 |
|------|------|------|
| `concept_hierarchy` | 领域知识 | `## 基础概念` → `## 核心算法` → `## 前沿扩展` |
| `workflow` | 工具操作 | `## 安装配置` → `## 日常操作` → `## 排障` |
| `comparison` | 方案评估 | `## 方案A路径` → `## 方案B路径` → `## 取舍` |
| `timeline` | 项目/学习 | `## Phase 1 验证` → `## Phase 2 扩展` → `## Phase 3 稳定` |

### Skill 层：note-crafter `### MOC 页面规范` 节重写

原 3 条机械约束 → 扩展为：设计原则（引用声明层）+ 分组策略表格（含反例）+ 机械约束（保留）。

### 设计决策

- **anti_example 仅 `concept_hierarchy` 配置**：其他三种策略的反例已隐含在表格中（`## 入门/高级`、`## 方案1/2`、`## 早期/后期`）。`concept_hierarchy` 需要额外标注 `## 杂项/其他` 这种常见偷懒写法。
- **不扩展 `forbidden_content`**：`## 杂项` 是分组质量问题，不是内容违规问题。放在 grouping_strategies 的 anti_example 中而非 forbidden_content。
- **声明层与 skill 层双写**：vault-rules.yaml 作为机器可读声明，note-crafter skill 作为人类可读指南。内容一致但表述不同。

## 影响范围

| 文件 | 变更 |
|------|------|
| `99-设置/vault-rules.yaml` | `moc` 块新增 `grouping_strategies` + `design_principles` |
| `~/.hermes/skills/note-taking/note-crafter/SKILL.md` | `### MOC 页面规范` 节重写 |

无向后不兼容变更。
