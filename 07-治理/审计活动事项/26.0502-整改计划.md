## 整改原则

- 先处理 P1，再处理 P2，P3 合并清理。
- 先修入口、MOC、断链，再做命名与内容质量优化。
- 不删除历史材料；确需归档时先保留可追溯入口。
- 未被规则覆盖的问题不强行整改，先记录为待确认事项。

## 分阶段计划

| 阶段 | 范围 | 退出标准 |
| --- | --- | --- |
| 第一阶段 | P1 修复 | 核心入口、MOC、引用链路恢复。 |
| 第二阶段 | P2 整改 | 命名、元数据、结构边界达到规则要求。 |
| 第三阶段 | P3 清理 | 低风险不一致被合并处理或明确豁免。 |

## 风险控制

| 风险 | 控制方式 |
| --- | --- |
| 误移动文件导致断链 | 先检索引用，再调整路径，最后复查反链。 |
| MOC 变成进度台账 | MOC 只保留入口链接，执行记录放整改报告。 |
| 规则边界不清 | 标记为待确认，不纳入强制整改。 |

## 验收标准

- P1 项全部关闭。
- P2 项有明确处理结果：已修复、延期、豁免。
- MOC 已挂载审计报告、整改计划、整改报告。
- 验证命令或人工复查结果已写入整改报告。
