# 术语规范审计报告

**日期**：2026-05-02
**编码**：术语规范-26.0502-001
**对象**：`99-设置/specification/术语审计规范/术语审计规范.md`
**关联规则源**：`99-设置/vault-rules.yaml`、`99-设置/agent-skills/note-crafter/core.md`
**关联执行层**：`03-领域/00-概念术语/` 下术语卡与 Base 数据库

## 结论

术语规范当前最大问题不是规则缺失，而是规则源之间尚未完成收敛：声明层和 note-crafter 已明确新术语笔记使用 `type: note`，但术语审计规范仍把 `term` 与 `note` 作为待选择策略，现有 Base 又大量依赖 `type == "term"`。这会导致新增术语卡、旧术语卡、Base 检索和审计口径互相冲突。

建议以 `99-设置/vault-rules.yaml` 为最终声明层，采用 `type: note` 作为术语笔记统一口径；术语身份改由 `tags`、`aliases`、路径和必要的 `category/domain` 字段共同表达。随后再同步改写 Base 筛选条件。

## P1 — 规则源未统一

### [P1-1] `type` 字段口径冲突

当前存在三套口径：

| 规则源 | 口径 |
| --- | --- |
| `99-设置/vault-rules.yaml` | `terminology.frontmatter_type: note` |
| `99-设置/agent-skills/note-crafter/core.md` | 术语笔记固定使用 `type: note` |
| `术语审计规范.md` | 要求在 `term` 与 `note` 中择一，尚未固化 |
| 术语 Base | 多处使用 `type == "term"` |

**影响**：新增术语笔记会按 `type: note` 创建，但旧 Base 仍按 `type == "term"` 收录，导致新卡片漏收；若继续按 `term` 修复旧卡，又会反向违背声明层和 note-crafter。

**建议**：将术语审计规范改为确定口径：术语笔记统一 `type: note`；术语身份由 `tags` 或 `category/domain` 表达；Base 筛选条件同步从 `type == "term"` 改为路径 + 标签/字段组合。

### [P1-2] Base 筛选依赖旧枚举

`03-领域/00-概念术语/` 下 Base 检出 12 处 `type == "term"`。这与声明层 `frontmatter_type: note` 不一致。

**影响**：术语数据库无法稳定收录新术语卡。审计报告、整改计划和实际视图会给出不同结果。

**建议**：先冻结统一筛选策略，再批量改 Base。建议策略为：

```text
路径属于术语库
AND type == "note"
AND tags / category / domain 表达术语身份
```

## P2 — 执行层与规范层仍有旧结构残留

### [P2-1] 通用术语目录 Frontmatter 枚举混用

`03-领域/00-概念术语/` 三层内 Markdown 的 `type` 统计如下：

| type 值 | 数量 |
| --- | ---: |
| 空值 | 30 |
| `note` | 11 |
| `term` | 27 |
| `术语卡片` | 2 |
| `概念定义` | 1 |

**影响**：术语卡无法通过单一字段稳定筛选，后续迁移和 Base 维护成本上升。

**建议**：先修规范，再修执行层。规范未定前，不建议继续局部补 `type: term`。

### [P2-2] Base 仍引用旧路径或临时路径

本轮复核检出代表性旧路径：

| Base | 旧路径 |
| --- | --- |
| `@db-通用术语数据库(db@common-terms).base` | `---/术语库(Terms svy)/01-一般术语知识库(Common DB)` |
| `@db-通用术语数据库(db@common-terms).base` | `03-领域/60-艺术学科/00-艺术术语库` |
| `db-术语数据库(db@ai terms).base` | `---/--数据库看板/一般术语文件柜` |
| `db-术语数据库(db@ai terms).base` | `03-领域/60-艺术学科/-艺术术语库(db@art-terms)` |
| `db-术语数据库(db@ai terms).base` | `---/--数据库看板/技术术语(db@Techinal terms)` |

**影响**：即使 `type` 口径统一，Base 仍可能漏收或指向失效位置。

**建议**：按现行 `vault-rules.yaml` 的 `terminology.general_root`、`general_default` 和 `discipline_rule` 重写路径。

## P3 — 规范文本自身可读性改进

### [P3-1] 验收标准缺少“规则源同步”项

现有整改验收包含入口、Base、术语卡、断链和非术语数据库清理，但没有明确要求同步更新：

- `vault-rules.yaml`
- `术语审计规范.md`
- note-crafter / hxc-term-card 等创建规则
- Base 筛选条件

**影响**：单次整改可能修好了文件，却没有修好后续创建规则，问题会反复出现。

**建议**：在术语审计规范的“整改验收”中加入“规则源一致性”验收项。
