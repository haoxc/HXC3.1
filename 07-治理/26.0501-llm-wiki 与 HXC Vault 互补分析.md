# llm-wiki 与 HXC Vault 互补分析

**结论**：llm-wiki 和 HXC Vault 不是替代关系，而是「深度编译」和「广度容器」的互补。HXC Vault 管个人全部知识（多域混杂），llm-wiki 管单一课题的深度交叉引用（Agent 编译产物）。

**理由**：
1. 组织维度不同——llm-wiki 按类型扁平（entities/concepts/comparisons），HXC 按功能层级（领域/工具/输入）。强行统一制造类别错误
2. 适用知识阶段不同——碎片摘录放 HXC 的 `02-输入/`，需要交叉引用和矛盾管理时才升级为 wiki
3. `---/` 下的课题天然是 llm-wiki 场景——边界清晰、多源输入、需要 Agent 自动交叉引用

**出处**：llm-wiki SKILL.md v2.2.0, pjx-common-auditor SKILL.md v2.7.0, 2026-05-01 评估会话。

## 兼容性矩阵

| 维度   | llm-wiki                                                       | HXC Vault                         | 兼容性          |
| ---- | -------------------------------------------------------------- | --------------------------------- | ------------ |
| 组织轴  | 类型扁平（entities/concepts/raw/）                                   | 功能层级（01-我的/ 03-领域/ 09-工具/）        | 互斥           |
| 导航面  | index.md 分区目录                                                  | MOC 文件夹笔记                         | 概念不对等        |
| 元数据  | title/created/updated/type/tags/sources + confidence/contested | tags/description/type/create-date | 互斥           |
| 文件命名 | en-lowercase-hyphens                                           | 中文名词                              | 互斥           |
| 标签治理 | SCHEMA.md 白名单（硬约束）                                             | vault-rules.yaml 推荐词表（软约束）        | 可抽象共享        |
| 链接规则 | Wikilink，≥2 outbound links/page                                | Wikilink，≥1 发现路径                  | 可直接迁移        |
| 来源追溯 | provenance markers + sha256 漂移检测                               | description 字段                    | llm-wiki 更严格 |
| 操作日志 | log.md append-only，500 条轮转                                     | 07-治理/ 日期前缀文件                     | 形式不同         |

## 使用场景

### llm-wiki 适用

- 单一主题深度研究（「AI Agent 全景」「Crypto 监管政策」）
- Agent 主导的知识编译（论文→实体/概念→交叉引用）
- 需要显式矛盾管理（contested/contradictions）
- 来源可追溯是核心需求（provenance markers）
- 知识需要复利（第 50 篇论文自动链接前 49 篇的实体）

### HXC Vault 适用

- 个人日记/周报/月报
- 工具使用笔记（按工具名检索 > 按实体检索）
- 多域混杂（AI + 设计 + 商业 + 个人）
- 人类自己写 > Agent 写
- 临时想法、碎片、半加工内容
- 项目管理（需求/方案/决策/交付）

### 知识阶段迁移

```
阅读摘录          深度研究          知识固化
   │                │                │
   ▼                ▼                ▼
02-输入/    →    ---/课题/    →   03-领域/
(碎片归档)     (llm-wiki编译)    (成型知识笔记)
```

一个知识域在「阅读摘录」阶段放 `02-输入/`，在「需要交叉引用、矛盾管理、来源追溯」时升级为 `---/` 下的 llm-wiki，完成后核心结论提炼到 `03-领域/` 做长期知识沉淀。

## 课题级嵌入模式

`---/` 下的课题可直接按 llm-wiki 三层架构组织：

```
---/领域-智能体调研/
├── SCHEMA.md        # 课题领域定义 + 标签分类
├── index.md         # 内容目录
├── log.md           # 操作追溯
├── raw/             # 论文/文章原文（不可变）
├── entities/        # 关键产品/公司/模型
├── concepts/        # 方法论/概念
├── comparisons/     # 方案对比
└── queries/         # 有价值的研究问答
```

**与父 Vault 的交叉引用**：
- wiki 内部用 `[[wikilinks]]`
- 父 Vault 笔记链入：`[[领域-智能体调研/entities/langchain.md]]`
- wiki 链出：`[[../../03-领域/AI/RAG 架构]]`

## 审计分工

| 审计对象 | 审计工具 | 维度 |
|----------|---------|------|
| HXC Vault 主体（01/ 02/ 03/ ...） | pjx-common-auditor | KV-1 ~ KV-11 |
| `---/` 下 llm-wiki 课题 | llm-wiki lint | 12 项（orphans, broken links, index, frontmatter, stale, contradictions, quality, drift, size, tags, log, report） |
| 课题与 vault 的交叉引用 | pjx-common-auditor | KV-9（断链检测） |

pjx 检测到三要素签名（SCHEMA.md + raw/ + entities/）时自动委托 llm-wiki lint，不套用 KV 维度。
