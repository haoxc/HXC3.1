## 一、项目概述

### 1.1 项目背景

随着数字化转型的深入推进，企业对数字孪生技术的应用需求日益增长。数字孪生技术通过构建物理世界的数字镜像，实现设施、设备、IoT物联网等资产的全生命周期管理。本项目旨在搭建一个统一的数字孪生数据中台，为3D可视化场景提供标准化的数据支撑能力。

### 1.2 中台定位

本数据中台定位为**纯数据层**，专注于以下核心职能：

- **数据对接**：统一接入各类业务系统、IoT设备、BIM/GIS等数据源
- **数据治理**：提供数据质量管理、标准化、血缘追溯等能力
- **数据审计**：完整记录数据变更历史，支持合规审计
- **数据服务**：为3D渲染引擎等上层应用提供标准化API接口

> [!important] 边界说明
> 中台不负责3D渲染，仅提供数据接口供ThreeJS等渲染引擎消费。

### 1.3 技术选型

| 技术领域   | 选型方案              | 说明                              |
| ---------- | --------------------- | --------------------------------- |
| 后端语言   | Java (Spring Boot)    | 生产环境已采用，团队熟悉          |
| 数据库     | PostgreSQL            | 关系数据 + PostGIS空间 + TimescaleDB时序 |
| 前端框架   | Vue3 + Element Plus   | 管理后台界面                      |
| 3D引擎     | ThreeJS               | 由上层应用集成，中台仅提供数据    |
| IoT网关    | EMQX                  | MQTT/OPC-UA协议适配               |
| 模板引擎   | Amis (百度开源)       | 低代码页面渲染，稳定成熟          |
| 对象存储   | MinIO                 | 图元文件、3D模型存储              |

## 二、总体架构设计

### 2.1 架构分层

![[architecture-overview.svg]]

数字孪生数据中台采用分层架构设计，自上而下分为：

1. **外部消费层**：3D渲染引擎、业务系统、移动端/大屏、第三方集成
2. **API网关层**：统一接口服务、鉴权、限流、路由
3. **核心服务层**：接口管理、标识管理、图元管理、模板引擎
4. **孪生体领域层**：孪生体模型、空间拓扑、生命周期管理
5. **数据治理层**：数据质量、数据标准、数据血缘、数据审计
6. **数据接入层**：IoT适配器、API适配器、文件适配器、消息适配器
7. **数据存储层**：PostgreSQL关系数据、TimescaleDB时序、PostGIS空间、MinIO文件

### 2.2 边界说明

| 职责范围 | 中台负责                       | 中台不负责             |
| -------- | ------------------------------ | ---------------------- |
| 3D渲染   | 提供孪生体数据、图元资源URL    | 场景渲染、交互逻辑     |
| IoT数据  | 协议适配、数据存储、订阅推送   | 边缘计算、设备控制指令 |
| 业务逻辑 | 数据标准化、API封装            | 业务流程、审批工作流   |

## 三、接口管理中心

### 3.1 功能概述

接口管理中心提供可视化方式统一管理业务系统接口，支持接口的注册、分类、查询、测试及运行时联调功能。

![[api-management-lifecycle.svg]]

### 3.2 核心功能

#### 3.2.1 接口注册与发现

- 支持OpenAPI/Swagger文档自动导入，解析接口元数据
- 支持手动配置接口，填写URL、方法、参数Schema等
- 自动发现已注册接口的版本变更，提示更新

#### 3.2.2 接口分类与标签

- **按业务域分类**：设备管理、告警中心、资产管理、生产运营等
- **按数据类型分类**：实时数据、历史数据、配置数据、统计数据
- **按孪生体类型分类**：设施、设备、传感器、区域等
- 支持自定义标签，灵活扩展分类维度

#### 3.2.3 在线测试

- 类Postman的可视化测试界面
- 支持参数模板、环境变量配置
- Mock数据支持，便于前端独立开发
- 请求/响应历史记录，支持回放

#### 3.2.4 运行时联调

- **实时流量监控**：QPS、延迟、错误率仪表盘
- **链路追踪**：集成Jaeger/SkyWalking，可视化调用链
- **联调沙箱**：多环境隔离，开发/测试/生产分离
- **熔断限流**：保护策略配置，防止级联故障

### 3.3 数据模型

**api_registry表**：

| 字段名          | 类型          | 说明                        |
| --------------- | ------------- | --------------------------- |
| api_id          | UUID          | 接口唯一标识                |
| name            | VARCHAR(200)  | 接口名称                    |
| url             | VARCHAR(500)  | 接口地址                    |
| method          | ENUM          | HTTP方法：GET/POST/PUT/DELETE |
| version         | VARCHAR(20)   | 接口版本号                  |
| status          | ENUM          | 状态：草稿/已发布/已下线    |
| category        | VARCHAR(100)  | 业务分类                    |
| twin_type       | VARCHAR(100)  | 关联孪生体类型              |
| request_schema  | JSONB         | 请求参数Schema定义          |
| response_schema | JSONB         | 响应结果Schema定义          |
| auth_type       | ENUM          | 认证方式：无/Token/OAuth2   |
| timeout         | INT           | 超时时间(毫秒)              |
| mock_enabled    | BOOLEAN       | 是否启用Mock                |
| mock_data       | JSONB         | Mock响应数据                |

## 四、标识与标签系统

### 4.1 设计目标

构建统一的标识体系，支持多行业标准编码（KKS、IEC 61850、CIM等），实现孪生体在规划阶段的位置、属性等信息的标准化管理。

![[identity-system.svg]]

### 4.2 标识编码体系

#### 4.2.1 KKS编码（主标识）

KKS（Kraftwerk-Kennzeichensystem）是电力行业设备标识系统，作为本平台的**主标识体系**。

- **编码结构**：电厂标识 + 系统标识 + 设备标识 + 元件标识
- **示例**：`MAA10CT001`（主变压器A相电流互感器）
- 支持KKS编码的自动解析与校验

#### 4.2.2 IEC 61850标识

支持IEC 61850变电站通信标准的设备标识：

- 逻辑节点（LN）标识映射
- 数据对象（DO）与数据属性（DA）关联
- 与KKS编码的双向映射关系维护

#### 4.2.3 其他编码体系

- **CIM/BIM编码**：建筑信息模型标识，IFC GUID
- **自定义编码**：企业内部标准，如AREA-FLOOR-DEV格式
- **扩展预留**：支持后续接入新的行业标准

### 4.3 标签系统

| 标签类型   | 用途                           | 示例                           |
| ---------- | ------------------------------ | ------------------------------ |
| 位置标签   | 规划坐标、实际坐标、楼层、区域 | x:100,y:200,z:50; Floor:3      |
| 类型标签   | 设备类型、设施类型、传感器类型 | Transformer; Sensor:Temperature |
| 业务标签   | 责任人、合同编号、项目归属     | Owner:张三; Contract:C2024001  |
| 状态标签   | 运行状态、告警状态、维护状态   | Status:Running; Alert:Warning  |
| 自定义标签 | 用户自定义的扩展属性           | Priority:High; Batch:2024Q1    |

### 4.4 数据模型

**twin_identity表**：

| 字段名      | 类型          | 说明                             |
| ----------- | ------------- | -------------------------------- |
| identity_id | UUID          | 标识记录ID                       |
| twin_id     | UUID FK       | 关联孪生体                       |
| code_system | ENUM          | 编码体系：KKS/IEC61850/CIM/CUSTOM |
| code_value  | VARCHAR(200)  | 编码值                           |
| is_primary  | BOOLEAN       | 是否主标识                       |
| valid_from  | TIMESTAMP     | 生效时间                         |
| valid_to    | TIMESTAMP     | 失效时间                         |

## 五、图元管理

### 5.1 功能概述

图元管理模块负责孪生体显示图元信息的统一管理，包括2D图标、3D模型、动画效果等视觉资源的存储、分类、版本控制及与孪生体的绑定关系。

![[visual-template-system.svg]]

### 5.2 图元仓库

#### 5.2.1 支持的图元类型

| 图元类型 | 文件格式        | 适用场景               |
| -------- | --------------- | ---------------------- |
| 2D图标   | SVG/PNG         | 平面图、流程图、状态指示 |
| 3D模型   | glTF/GLB/OBJ    | 三维场景渲染           |
| 动画效果 | Lottie/Spine    | 告警动画、状态变化动效 |
| 材质贴图 | PNG/JPG/HDR     | 3D模型材质             |

#### 5.2.2 LOD层级管理

支持多层次细节（Level of Detail）管理，适配不同渲染场景的性能需求：

- **LOD0（高精度）**：近距离观察，完整细节
- **LOD1（中精度）**：中等距离，简化细节
- **LOD2（低精度）**：远距离，基础轮廓
- **LOD3（图标）**：极远距离，点状图标替代

### 5.3 孪生体-图元绑定

- **按类型绑定**：同类型孪生体共享图元模板
- **按状态切换**：正常/告警/故障/维护等状态对应不同图元样式
- **按场景适配**：不同可视化场景（巡检/监控/展示）使用不同图元
- **个性化覆盖**：支持单个孪生体覆盖默认图元配置

## 六、模板渲染引擎

### 6.1 设计目标

基于Amis低代码平台，实现数据与渲染机制、样式的解耦，通过页面模板配置化方式，根据孪生体关联的业务信息动态渲染展示页面。

### 6.2 三层解耦架构

| 层级   | 职责                               | 技术实现                   |
| ------ | ---------------------------------- | -------------------------- |
| 数据层 | 孪生体属性数据、IoT实时数据、业务数据 | API接口、WebSocket推送     |
| 逻辑层 | 数据转换规则、条件渲染逻辑、事件响应 | Amis表达式、自定义适配器   |
| 展示层 | 页面模板、样式主题、布局配置       | Amis JSON Schema           |

### 6.3 核心功能

#### 6.3.1 模板设计器

- 基于Amis Editor的可视化拖拽设计
- 丰富的组件库：图表、表格、表单、卡片等
- 模板版本管理与发布流程
- 模板预览与调试工具

#### 6.3.2 数据绑定机制

- **字段映射**：孪生体属性与模板字段的绑定关系
- **数据转换**：格式化、单位换算、条件判断
- **实时数据**：WebSocket订阅IoT数据自动刷新
- **聚合计算**：多数据源联合查询与计算

#### 6.3.3 动态模板切换

根据孪生体类型、状态、上下文自动匹配最佳展示模板：

- **设备详情页**：展示设备基本信息、实时参数、历史趋势
- **告警面板**：告警详情、处置建议、关联设备
- **运维工单**：工单信息、执行步骤、历史记录
- **统计报表**：汇总数据、图表分析、导出功能

### 7.1 实体关系概览

![[data-model-er.svg]]

- **twin_entity**：孪生体实体主表
- **twin_type**：孪生体类型定义（支持继承）

## 八、数据治理

### 8.1 数据质量管理

- **数据清洗**：异常值检测、缺失值处理、重复数据识别
- **数据校验**：基于规则的自动校验，Schema约束
- **质量评分**：完整性、准确性、一致性、及时性多维度评分
- **质量报告**：定期生成数据质量分析报告

### 8.2 数据血缘（分阶段）

![[data-lineage-assessment.svg]]

> [!tip] 推荐方案：分阶段演进

#### Phase 1：PostgreSQL邻接表

- 使用关系表存储血缘关系
- WITH RECURSIVE查询实现多跳追溯
- 适用场景：3跳以内的血缘查询
- 优势：无新增组件，事务一致性好

```sql
-- 血缘表结构
CREATE TABLE data_lineage (
    id UUID PRIMARY KEY,
    source_type VARCHAR(50),
    source_id UUID,
    target_type VARCHAR(50),
    target_id UUID,
    relation VARCHAR(50),
    transform_rule JSONB,
    created_at TIMESTAMP
);

-- 递归查询下游影响
WITH RECURSIVE downstream AS (
    SELECT * FROM data_lineage WHERE source_id = :entity_id
    UNION ALL
    SELECT l.* FROM data_lineage l
    JOIN downstream d ON l.source_id = d.target_id
)
SELECT * FROM downstream;
```

#### Phase 2：评估扩展（按需）

- Apache AGE扩展：为PostgreSQL增加图查询能力
- 独立Neo4j：复杂图分析场景
- 评估触发条件：查询性能不足或需要复杂图算法

#### Phase 3：全链路血缘（远期）

- Apache Atlas：跨系统元数据血缘管理
- 适用于大数据生态、多系统集成场景

### 8.3 数据审计

- **变更日志**：记录谁在何时改了什么
- **操作追踪**：CRUD操作全记录
- **快照备份**：关键时间点数据状态保存
- **合规报告**：支持审计报告导出
