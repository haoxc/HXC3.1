# 课题-知识分离底层逻辑与落地规划

**结论**：课题层与知识层分离的核心判据是「读者能否脱离过程上下文独立理解」。知识层反向引用课题层会污染稳定知识，让 MOC 从知识地图退化为项目进度入口。当前设计完整、实施为零——声明层缺 workspaces 节、审计不检测反向引用、note-crafter 不区分三路路由。

## 底层逻辑

### 核心判据

| 判据 | 课题层 | 知识层 |
|------|--------|--------|
| 读者需要什么上下文 | 需要了解这个课题的来龙去脉 | 只需要知道这个概念本身 |
| 内容是否仍在变化 | 频繁变化、有争议、有临时判断 | 多次验证稳定、可独立成立 |
| 有效期 | 与课题生命同周期 | 长期，直到被更好解释替代 |

### 知识层不能反向引用课题层的三个理由

**1. 污染稳定知识**

知识层的读者不应被迫了解某个课题的过程才能理解一个概念。反向引用在正文中出现「详见 [[课题-xxx]]」时，读者需要跳转到过程文档才能获得完整理解——这破坏了知识层的自包含性。

| ✓ 正确 | ✗ 错误 |
|--------|--------|
| `## 转换矩阵\n转换矩阵将 A 空间向量映射到 B 空间。` | `## 转换矩阵\n详见 [[课题-UE引擎调研]] 第三章的讨论。` |

**2. MOC 从知识地图退化为项目进度入口**

知识层 MOC（如 `03-领域/50-管理学科/50-管理学科.md`）的职责是导航稳定知识。如果它的链接列表里混入课题过程页面，读者无法区分「这是成熟知识」还是「这是进行中的调研」——检索信任感崩塌。

**3. 一棵树一张嘴被破坏**

Canonical 页面原则要求每个概念只有一个主解释页。如果知识层页面通过反向引用把解释责任推给课题层，同一个概念就有了两个「半张嘴」——知识层页只说了定义、课题层页说了推导——读者需要跑两个地方才能拼出完整理解。

### 允许的例外

| 场景 | 允许形式 | 理由 |
|------|---------|------|
| 稳定知识需来源追溯 | 页末链课题入口（不链过程碎片） | 读者可以跳过，不影响正文理解 |
| 领域 MOC 提示研究方向 | MOC 独立一节「研究课题」 | 读者清楚这是待研究，不是成熟知识 |

> 仅链 `课题-{主题}.md`（课题入口），不链 `raw/`、`log/`、临时分析页。

## 落地规划

### Phase 1：声明层补缺（vault-rules.yaml）

在 `vault-rules.yaml` 新增 `workspaces` 节，使反向引用约束从「人类约定」升级为「机器可校验」。

```yaml
workspaces:
  path: "---/-workspaces/"
  role: project
  description: 课题编译层——过程、材料、争议、临时判断
  domain_reference:
    rule: "知识层（03-领域/06-商业/09-工具/05-资源）不应反向引用 ---/-workspaces/ 下的课题过程内容"
    exceptions:
      - "页末可链课题入口（仅课题 MOC，不链过程碎片）"
      - "领域 MOC 可设独立「研究课题」节，链主题目录入口"
    audit: "扫描所有 03-/06-/09-/05- 目录下 .md 的 Wikilink，若目标在 ---/-workspaces/ 且非课题 MOC → 违规"
  organization:
    four_zones: ["元管理", "主题课题群", "组织课题群", "根层过渡"]
    naming:
      topic_dir: "{主题}课题/"
      project_dir: "课题-{主题}({日期})/"
    subdir: semantic
```

### Phase 2：审计脚本增强（pjx-common-auditor）

新增检测项：

```
检查：知识层 → 课题层的反向 Wikilink
逻辑：
  1. 扫描 03-领域/、06-商业/、09-工具/、05-资源/ 下所有 .md
  2. 提取所有 [[...]] Wikilink
  3. 判断链接目标是否在 ---/-workspaces/ 下
  4. 如果在 → 检查是否为课题 MOC（文件名以 课题- 开头且非 raw/log/临时分析页）
  5. 非 MOC → 标记违规
输出：违规文件列表 + 链接行号
```

### Phase 3：note-crafter 三路路由

在 note-crafter skill 中新增路由分支（修改 `core.md` + `manifest.yaml`）：

```
Step 2 路由判断扩展为三路：

1. 课题管理？ → 课题层（课题说明/进展/审查）
   判据：笔记主题含「课题说明」「进展」「决策」「审查」关键词

2. 当前目标话题？ → 课程层
   判据：查课题台账清单获取当前目标话题，subagent 做语义匹配
   若漂移 → clarify 确认

3. 通用知识？ → 知识层
   走现有 vault-index.yaml 路由
```

### Phase 4：漂移检测 subagent

嵌入 note-crafter Step 2.5，用 delegate_task + 小模型做单次判断：

```
输入：目标话题描述 + 候选笔记摘要
输出：0-10 匹配度 + 理由 + 备选话题
阈值：≥7 无漂移 / 4-6 轻度标注 / ≤3 触发 clarify
```

### 依赖关系

```
Phase 1（声明层补缺）
  └─→ Phase 2（审计脚本）依赖声明层的新规则定义
        └─→ Phase 3（note-crafter 路由）依赖审计能验证路由正确性
              └─→ Phase 4（漂移检测）依赖三路路由已工作
```
