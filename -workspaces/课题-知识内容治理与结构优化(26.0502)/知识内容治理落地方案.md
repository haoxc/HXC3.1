# 知识内容治理落地方案

**结论**：采用“双层结构”：领域沉淀层用 HXC Vault，课题编译层吸收 Karpathy/llm-wiki 思想。不要把所有领域目录改成 wiki，也不要把深度课题继续塞进领域目录。

## 目标结构

### 领域沉淀层

用于长期、稳定、可检索的领域知识。

```text
03-领域/50-管理学科/
├── 50-管理学科.md
├── --管理术语表/
├── 01-管理-辨析与摘要/
├── 03-管理-方法工具/
└── 04-管理-知识/
```

治理规则：

- `50-管理学科.md` 是领域 MOC，只做导航和认知分组。
- `--管理术语表` 只放管理专用术语；跨领域通用术语进入全局术语库。
- `03-管理-方法工具` 放 FMEA、OKR、BSC、鱼骨图等方法工具。
- `04-管理-知识` 放已稳定沉淀的知识体系。

### 课题编译层

用于边界清晰、材料多、需要持续研究的课题。

```text
---/-workspaces/
├── 管理课题/
│   └── 课题-战略指标量化/
├── AI课题/
├── 教学课题/
└── ...
```

治理规则：

- 新课题按主题进入 `---/-workspaces/{主题}课题/`，如 `管理课题/`。
- 课题目录默认不加日期；创建前确认是否短期课题、长期课题或阶段性调研。
- 旧课题不批量迁移，只在继续推进、整改或复盘时单独评估。
- 原始资料进入 `raw/`。
- 关键对象进入 `entities/`。
- 方法、模型、原则进入 `concepts/`。
- 方案差异进入 `comparisons/`。
- 有复用价值的问题回答进入 `queries/`。

## 流转机制

```text
02-输入/
  ↓
---/-workspaces/{主题}课题/
  ↓
03-领域/50-管理学科/04-管理-知识/
```

含义：

- 阅读摘录和临时材料先进入 `02-输入/`。
- 需要深挖的内容进入 `---/-workspaces/{主题}课题/`。
- 研究完成后，只把稳定结论回流到 `03-领域/`。
- 领域层默认不反向引用课题过程内容；确需来源追溯时，只链接课题入口。

## 近期动作

1. 修复 `[[50-管理学科]]`：改为 `type: moc`，建立“术语 / 辨析 / 方法工具 / 知识体系 / 研究课题”五组导航。
2. 给 `经营维度`、`战略维度`、`指标设计` 指定 canonical 页面。
3. 将 `--/MOC` 中仍有价值的导航合并回根 MOC 或对应子目录 MOC。
4. 将深度课题从 `02-管理-课题` 中筛出，迁入 `---/-workspaces`。
5. 修复核心入口与课题入口中的未解析 wikilink。

## 收口原则

- MOC 是地图，不是正文。
- 领域目录沉淀结论，不保存全过程。
- 课题目录保存过程、来源和矛盾。
- 一个概念只保留一个主解释页，其他页面链接引用。
- 新建课题先确认主题目录、课题名称、日期策略和组织模式。
