# 课程层与课题层组织结构设计

**结论**：已有的双层架构（课题层 `---/-workspaces/` vs 知识层 `03-领域/`）解决了「过程留 workspace，结论回流领域」。本次向内推一层——每个课题内部拆为课题层（治理/项目管理）和课程层（与目标话题对齐的笔记产出），并配合课题台账清单 + 漂移检测机制，实现会话级课题驱动模式。

## 背景

当前 note-crafter 工作流假设所有笔记统一路由到知识层目录（`03-领域/`、`09-工具/` 等）。用户希望引入「课题驱动」模式——每次会话绑定一个目标话题，笔记创建时分三路判断：

1. 课题自身的管理笔记（课题说明/进展/审查）
2. 与目标话题对齐的学习/工作产出（课程层）
3. 通用知识笔记（知识层，走已有路由）

## 三路笔记路由决策树

```
笔记创建请求
  │
  ├─ 判断：是关于"课题管理"的吗？
  │   ├─ YES → 课题层（课题说明/进展/审查）
  │   └─ NO ↓
  │
  ├─ 判断：是关于"当前目标话题"的吗？ ← subagent LLM 语义判断
  │   ├─ YES → 课程层（课题-xxx/notes/）
  │   │   └─ 做漂移检测
  │   └─ NO ↓
  │
  └─ 通用知识 → 知识层路由
      └─ vault-index.yaml → 03-领域/ / 09-工具/ / 06-商业/ ...
```

## 课题台账清单

**位置**：`---/课题台账清单.md`，`type: moc`，作为所有课题的总索引。

### 字段

| 字段 | 类型 | 说明 | 示例 |
|------|------|------|------|
| 课题ID | `PROJ-{序号}` | 唯一标识，自动编号 | `PROJ-001` |
| 课题名称 | 文本 | 含主题+日期 | `课题-知识内容治理与结构优化(26.0502)` |
| 状态 | 枚举 | 活跃 / 暂停 / 完成 / 归档 | `活跃` |
| 分类 | 枚举 | 按治理对象分 | `知识管理` |
| 学科 | 枚举 | 关联学术/实践领域 | `管理学` |
| 关键词 | 逗号分隔 | 快速检索 + 去重检查 | `vault, note-crafter, 目录治理` |
| 目标话题 | 文本 | 本轮会话绑定的焦点 | `双层笔记架构设计` |
| 存放路径 | 相对 `---/-workspaces/` | | `项目治理课题/课题-知识内容治理与结构优化(26.0502)/` |
| 创建日期 | `YYYY-MM-DD` | | `2026-05-02` |
| 最近活动 | `YYYY-MM-DD` | 最近一次笔记创建/编辑 | `2026-05-02` |

### 状态判据

| 状态 | 判据 |
|------|------|
| 活跃 | 近 7 天有笔记产出或进展记录 |
| 暂停 | 无活动 7-30 天，或用户主动标记 |
| 完成 | 所有课程层笔记已产出；知识层回流已完成 |
| 归档 | 完成 ≥30 天，课题目录移入归档 |

## 单课题内部双层结构

借用已有的「知识内容治理双层」思路向内推一层：

```
课题-{主题}({日期})/
│
├── 📋 课题层（治理层）        ← "这个课题是什么、怎么管"
│   │
│   ├── 课题-{主题}.md         # MOC，整个课题的导航中枢
│   ├── 课题说明.md             # 章程：目标/范围/成功判据/关联知识域
│   ├── 课题进展.md             # 进度日志：决策记录/关键产出/待办
│   ├── 课题审查.md             # 漂移检测记录+审计
│   └── hermes/                 # Agent 工作物件（现有约定）
│       ├── plans/              # 执行计划
│       ├── decisions/          # 决策记录
│       └── logs/               # 会话日志
│
└── 📝 课程层（内容层）        ← "这个课题产出了什么"
    │
    ├── 课程笔记.md             # MOC，课程层导航
    └── notes/                  # 与目标话题对齐的笔记
        ├── {概念}.md
        ├── {工具}-{用途}.md
        └── {指南}.md
```

### llm-wiki 模式（可选覆盖）

当课题需要多源资料、实体/概念/对比/来源追溯时，课程层启用 llm-wiki 三要素：

```
课题-{主题}({日期})/
├── ...（课题层不变）
└── 课程层/
    ├── SCHEMA.md
    ├── raw/
    ├── entities/
    └── concepts/
```

## 三层判定对照

| 判定维度 | 课题层    | 课程层          | 知识层              |
| ---- | ------ | ------------ | ---------------- |
| 职责   | 项目怎么管  | 项目产出什么       | 知识永久沉淀在哪         |
| 读者   | 项目参与者  | 项目参与者+未来自己   | 任何人              |
| 生命周期 | 与课题同生死 | 课题结束后按转层标准评估 | 永久               |
| 漂移检测 | 不检测    | **检测**       | 不检测              |
| 转入条件 | 无      | 匹配目标话题       | 5 条件 + 10 分评分 ≥8 |

## 漂移检测机制

### 设计原则

用 subagent（小模型）做语义匹配，而非关键词匹配。理由：
- 关键词匹配误报率高（如「配置」一词同时出现在 hermes 和 gbrain 场景）
- 语义模型能判断「这篇笔记讨论的内容，在多大程度上与目标话题一致」

### 检测流程

```
1. 输入：目标话题描述（来自课题台账清单）+ 候选笔记内容摘要
2. subagent LLM 输出：
   - score: 0-10（语义匹配度）
   - reasoning: 一句话理由
   - suggested_topic: 如果漂移，建议的备选话题（可选）
3. 判断：
   - score ≥ 7 → 无漂移，创建到课程层
   - score 4-6 → 轻度漂移，创建到课程层 + 标注 flag
   - score ≤ 3 → 重度漂移，触发 clarify 确认
```

### clarify 模板（漂移时）

```
question: "笔记「{标题}」与当前课题「{目标话题}」语义匹配度较低（{score}/10）。
原因：{reasoning}
请选择操作："
choices:
  - "继续——仍放入当前课题的课程层"
  - "切换——改为课题「{备选课题名}」，并在台账中更新目标话题"
  - "放弃——不创建笔记"
```

### subagent 实现路径

通过 `delegate_task` 调用小模型做单次语义判断：

```
delegate_task(
  goal="判断笔记内容是否匹配目标话题",
  context="目标话题: {target_topic}\n笔记内容摘要: {summary}",
  toolsets=["terminal"],
  role="leaf"
)
```

小模型选择：`config.yaml` 的 `auxiliary_model` 或 `task_models` 配置低成本模型（如 deepseek-chat），单个判断 token <500。

## 与现有框架的对齐

| 已有设计            | 本次设计              | 对齐方式                        |
