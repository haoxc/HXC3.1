# 课题目录建构规则

**结论**：新课题原则上统一进入 `---/-workspaces/`，并按主题建立上层目录。领域层只沉淀稳定知识，不承载课题过程；是否从领域层链接回课题，需要按“是否服务理解、追溯或复盘”单独判断。

## 目录规则

```text
---/-workspaces/
├── 管理课题/
│   └── 课题-战略指标量化/
├── AI课题/
│   └── 课题-Agent技术能力边界/
├── 教学课题/
│   └── 课题-UE游戏引擎课程设计/
└── ...
```

规则：

- 所有新建课题默认进入 `---/-workspaces/{主题}课题/`。
- `{主题}` 由课题的主要治理对象决定，如管理、AI、教学、设计、项目治理。
- 旧课题不立即迁移；只有在需要整改、复盘或继续推进时，再单独评估迁移。
- 课题目录默认不加日期，因为创建时通常无法判断它是短期任务还是长期课题。
- 新建前必须确认：主题目录、课题名、是否加日期、是否按 llm-wiki 模式组织。

## 新建前确认项

| 确认项 | 默认值 | 需要追问的情况 |
| --- | --- | --- |
| 上层主题目录 | `---/-workspaces/{主题}课题/` | 主题归属不明确，或横跨多个主题 |
| 课题名 | `课题-{主题名}` | 标题过泛，如“管理优化”“AI方案” |
| 日期 | 默认不加 | 明确是会议、短期冲刺、阶段性调研 |
| 组织模式 | 普通课题目录 | 多源资料、需要实体/概念/对比/来源追溯时用 llm-wiki 模式 |
| 领域关系 | 不反向引用过程内容 | 需要知识来源追溯或深度阅读入口时，才链接课题入口 |

## 领域层引用边界

### 默认原则

领域层不应反向引用课题过程内容。原因是：

- 领域层承担知识沉淀，读者应能独立理解稳定结论。
- 课题过程包含材料、阶段判断、争议和临时上下文，容易污染稳定知识页。
- 领域层若频繁链接过程内容，会让 MOC 从“知识地图”退化成“项目进度入口”。

### 允许的例外

领域层可以链接课题，但只建议链接“课题入口”，不链接 raw、log、临时分析页。

| 场景 | 是否需要链接课题 | 理由 | 示例 |
| --- | --- | --- | --- |
| 稳定知识已可独立理解 | 不需要 | 领域层已完成沉淀，过程链接会增加噪音 | `指标设计-设计原则` 不必链接所有调研记录 |
| 结论仍需来源追溯 | 可以链接课题入口 | 便于回看形成过程，但不影响正文理解 | `指标设计原则` 页末加“形成来源：[[课题-战略指标量化]]” |
| 课题仍在推进，知识尚未稳定 | 不放入领域正文 | 避免未成熟判断污染知识层 | `课题-指标设计` 保留在 workspace |
| 领域 MOC 需要提示研究方向 | 可以放课题索引链接 | 只作为导航，不作为知识解释 | `50-管理学科.md` 中“研究课题”链接 `[[管理课题]]` |

## 风险评估

| 风险 | 表现 | 控制方式 |
| --- | --- | --- |
| 主题目录变成大杂烩 | `管理课题/` 下堆积过多无分组课题 | 建立主题 MOC，按状态、方向、知识回流目标分组 |
| 旧课题迁移破坏链接 | 原领域目录中的链接失效 | 旧课题不批量迁移；迁移前列源路径、目标路径、影响链接 |
| 不加日期导致时序不清 | 难以判断课题创建和演化阶段 | 使用 frontmatter `create-date`、`status`，必要时在阶段产物加日期 |
| 领域层与课题层断联 | 知识结论不知道来自哪里 | 只在需要追溯时链接课题入口，不链接过程碎片 |
| 课题重复创建 | 同一课题在多个主题目录出现 | 新建前搜索同名、近义词、aliases；必要时做合并说明 |
| 课题和知识边界再次混淆 | workspace 里沉淀知识，领域层又复制一份 | 用转层评分决定“提炼、合并、只索引、继续留课题层” |

## 执行口径

新建课题时，先输出一句确认：

```text
拟创建课题：---/-workspaces/{主题}课题/课题-{名称}/
日期策略：默认不加日期
组织模式：普通课题 / llm-wiki 课题
领域层关系：仅在稳定知识页需要追溯时链接课题入口
```

确认后再创建目录和 MOC。
