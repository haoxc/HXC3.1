# 目录规划通用原则（R6）

**结论**：Vault 目录规划遵循 7 条原则，按优先级排序为 归属优先 > 检索优先 > 一棵树一张嘴 > 课题-知识分离 > MOC 是地图 > 规模敏感 > 改动最小。前三条是结构原则（不可妥协），后四条是操作原则（场景权衡）。七条原则通过一个 N 叉决策流回答核心问题：「这个内容该放哪」。

## 论点 1：归属优先——内容先定顶层域，再定深度（优先级 1）

**判据**：内容能否被明确归入一个顶层目录？
- 能 → 先定归属再定深度
- 不能 → 退化为按主题临时放置 + 标记 `review_later`

**归属判断表**：

| 顶层目录 | 内容特征 | ✓ 示例 | ✗ 反例 |
|----------|----------|--------|--------|
| `03-领域/` | 学术/技术/方法论——跨项目可复用的知识 | `Transformer 架构` | `公司 A 的 Transformer 应用案例`（→ 06-商业/） |
| `06-商业/` | 公司/产品/市场/竞争——与具体实体绑定的情报 | `Notion 产品分析` | `笔记工具的通用设计原则`（→ 03-领域/） |
| `07-治理/` | Vault 自身的结构/规则/审计/维护 | `skill规范整改台账` | `项目管理方法论`（→ 03-领域/） |
| `---/` | 课题/工作区——过程产物，非稳定知识 | `课题-知识内容治理` | `知识管理最佳实践总结`（→ 03-领域/） |
| `01-我的/` | 个人日志/生活记录/非分享内容 | `日记-2026-05-02` | `Obsidian 配置技巧`（→ 03-领域/） |

**反例**：
- 跨域内容（如「AI 在商业中的应用」）→ 放主要归属域（03-领域/31-人工智能学科），在次要域用 Wikilink 引用，不复制
- 归属模糊内容（如「写作方法论」——是 03-领域/ 还是 01-我的/？）→ 如果方法是可推广的、有理论支撑的，归 03-领域/；如果纯粹是个人偏好，归 01-我的/

**论据**：归属错了，后续所有操作（MOC 挂载、索引、检索路径）都是错误基础上的修补。这是唯一一条「不可逆」成本最高的原则。

**材料**：`99-设置/vault-rules.yaml` top_level 节

## 论点 2：检索优先——文件名=核心主题词，不加冗余前缀（优先级 2）

**判据**：
1. 文件名是否等于核心主题词？
2. 标题是否在前 15 行内可扫描到核心结论？

**命名规则**：

| 规则 | ✓ | ✗ |
|------|---|---|
| 文件名=笔记标题 | `PostgreSQL 索引优化` | `关于 PostgreSQL 索引优化` |
| 不加「如何」「关于」「浅谈」 | `环境配置指南` | `如何配置开发环境` |
| 消歧义用竖线别名 | `Python\|Python 语言` | `Python（编程语言）` |
| 日期前缀仅课题目录 | `2026-05-02-项目名/` | `2026-05-02-PostgreSQL 索引优化.md` |

**反例**：
- 允许前缀的场景：同名不同义消歧义（`Python|Python 语言` vs `Python|Monty Python`）
- 不冗余不等于不精确：「索引优化」比「数据库性能」更精确，选前者

**论据**：搜不到 = 不存在。Obsidian Quick Switcher 按文件名匹配，标题决定搜不搜得到。

**材料**：[[CLAUDE.md]] 标题规则 + 文件命名规则

## 论点 3：一棵树一张嘴——每个概念只有一处 canonical 位置（优先级 3）

**判据**：
1. 两个页面是否在解释完全相同的概念？
2. 如果是全等解释 → 合并到更完整的页面
3. 如果有差异侧重 → 保留 + 互链，不做合并

**操作表**：

| 发现场景 | 处理方式 |
|----------|----------|
| 两个页面全等解释同一概念 | 合并，保留更完整的那页，另一页改为 Wikilink 重定向 |
| 两个页面解释同一概念但侧重不同 | 保留两页，互相 `[[链接]]` + 注明侧重点差异 |
| 同一概念在 5 个页面各解释一段 | 提取为 canonical 页面，原位置改为 Wikilink |

**反例**：
- 不合并的场景：`Transformer 架构` 和 `Transformer 在推荐系统中的应用`——后者是前者的应用实例，不是对概念本身的重复解释
- 不提取的场景：日记中的一段理解——那是个人上下文中的临时记录，不是 canonical 解释

**论据**：每多一处重复解释，就多一个需要同步更新的点。修改时遗漏一处 = 信息不一致。

**材料**：[[CLAUDE.md]] 底层逻辑第 3 条；[[管理学科高频概念canonical页面清单]]

## 论点 4：课题-知识分离——过程产物和稳定输出不在同一层（优先级 4）

**判据**：
1. 内容是过程产物（还在迭代、依赖上下文）？→ 课题层 `---/-workspaces/`
2. 内容是稳定结论（可脱离原始材料独立理解）？→ 评估转入领域层

**分离表**：

| 属性 | 课题层 | 领域层 |
|------|--------|--------|
| 稳定性 | 迭代中，可能修改 | 稳定，修改需评估影响 |
| 独立性 | 依赖课题上下文 | 可脱离上下文独立理解 |
| 时效性 | 项目结束即归档 | 长期有效 |
| 目录 | `---/-workspaces/课题名/` | `03-领域/` `06-商业/` `07-治理/` |
| type | note | note |

**反例**：
- 短期临时笔记（todo/草稿/会议记录）→ 日记 `## 捕获` 节即可，不需要单独建课题
- 课题已完成但产出未提炼 → 课题归档，标记 `status: completed; needs_transfer`，由 Vault Steward 后续处理

**论据**：混杂导致两个问题——检索时被过程笔记淹没（信噪比下降），以及稳定知识被课题上下文绑定而无法被独立发现。

**材料**：[[课题知识转入领域知识的判定标准]]；[[HXC Vault与Karpathy知识组织对比]]

## 论点 5：MOC 是地图——不做文章，只做导航（优先级 5）

**判据**：
1. MOC 正文 ≤20 行？
2. MOC 链接数 ≤40？
3. 是否有段落式展开说明？→ 有则违反，应提取为独立笔记

**规则**：

| 规则 | ✓ | ✗ |
|------|---|---|
| MOC 正文只写导航说明 | `本节涵盖以下主题：` + 链接列表 | 在 MOC 中展开解释每个概念 |
| 领域总览用 type: note | `人工智能概述.md` type=note | 把概述写成 MOC 让链接列表被淹没 |
| 超过 40 链接考虑拆分 | 拆为子 MOC 或按主题分组 | 不拆，让 MOC 变成 60 个链接的列表页 |

**反例**：
- 领域总览页（如 `人工智能概述`）可以超过 20 行——它是文章不是 MOC，应标记 type: note
- 分组清晰的 MOC（如按字母序的工具列表）即使 >40 链接也可以不拆——扫描效率没降低

**论据**：MOC 的价值是「快速定位」，不是「深度阅读」。把文章内容混入 MOC 会打断导航流。

**材料**：`99-设置/vault-rules.yaml` moc 节

## 论点 6：规模敏感——量化阈值触发拆分决策（优先级 6）

**判据**（来自 vault-rules.yaml）：

| 对象 | 阈值 | 动作 |
|------|------|------|
| 单篇笔记 | >80 行 | 考虑拆成子页面，用 `[[子页面]]` 链接引出 |
| MOC | >40 链接 | 考虑拆分为子 MOC 或按主题分组 |
| 目录 | >500 note | 触发 Topic Architect 评估拆分方案 |

**触发后流程**：
```
阈值触发 → Topic Architect 评估
  ├─ 内聚度高、拆后跨 MOC 链接反而增加 → 不拆，记录豁免
  ├─ 子主题不稳定（仍在快速演化）→ 标记监控，推迟决策
  ├─ 拆分导致 canonical 需跨子目录引用 → 评估引用成本 > 拆分收益？
  └─ 拆分合理 → 出方案 → 用户确认 → Vault Steward 或 Note Crafter 执行
```

**反例**：
- 目录虽 >500 但内容高度内聚（如 `03-领域/31-人工智能学科/` 下的模型笔记）→ 先看是否可以分子主题 MOC 而非分子目录
- 单篇笔记 85 行但结构完整、读者不滚动也能看到结论 → 不拆

**论据**：阈值是信号不是命令。超过阈值触发评估，评估结论可以是「不拆」。

**材料**：`99-设置/vault-rules.yaml` threshold 节

## 论点 7：改动最小——优先修补入口而非迁移存量（优先级 7）

**判据**：当目录结构需要调整时，计算两条路径的成本：
1. 修补方案：只改入口（MOC 链接、索引路径）+ 新增内容遵循新规则
2. 搬迁方案：大量迁移现有文件 + 修复所有 Wikilink

**选择**：优先 1，除非不改动的长期代价超过一次性搬迁成本。

**三原则**（来自管理学科整改计划）：

| 原则 | 含义 |
|------|------|
| 不一次性大搬家 | 迁移分批次，每批验证后再下一批 |
| 先修入口再迁移 | 先确保 MOC/索引指向正确位置，再移动文件本身 |
| 旧页面不删除先归档 | 迁移后旧页保留并标记 `archived: true`，确认新页稳定后再删除 |

**反例**：
- 当不改动的代价超过改动时（如整目录已腐烂、每次找文件都要手动翻）→ 允许大改，但必须：完整评估 → 一次做完 → 全量验证 → 记录决策

**论据**：这是一条兜底原则。当其他 6 条原则产生冲突时，改动最小让位于前三条结构原则，但作为操作约束确保不产生「为优化而优化」的无效搬迁。

**材料**：[[50-管理学科整改计划]]

## 7 原则决策流（N 叉树）

回答「这个内容该放哪」，按顺序过决策节点：

```
新内容
  │
  ├─ 归属判断（原则 1）
  │   ├─ 03-领域/ → 定子域 → 下钻到具体目录
  │   ├─ 06-商业/ → 定公司/领域 → 下钻
  │   ├─ 07-治理/ → 定子主题 → 下钻
  │   ├─ ---/ → 课题-知识分离判断（原则 4）
  │   │   ├─ 临时/过程 → 课题目录
  │   │   └─ 稳定结论 → 回到归属判断，入领域层
  │   ├─ 01-我的/ → 日记/个人记录
  │   └─ 模糊 → 就近主题目录 + review_later 标记
  │
  ├─ 命名确认（原则 2）→ 文件名=核心主题词
  │
  ├─ 去重检查（原则 3）→ 已有 canonical 页面？
  │   ├─ 有且全等 → 合并/重定向
  │   ├─ 有但侧重不同 → 保留 + 互链
  │   └─ 无 → 新建
  │
  ├─ MOC 挂载（原则 5）→ 挂到哪个 MOC？新 MOC 链接数是否 >40？
  │
  ├─ 规模检查（原则 6）→ 目标目录是否 >500？是否会触发 MOC >40？
  │
  └─ 改动评估（原则 7）→ 本次操作需要迁移多少已有内容？
```

## 优先级与冲突解决

| 冲突组合 | 判决 | 理由 |
|----------|------|------|
| 归属(1) vs 规模(6) | 归属赢 | 归属正确优先于目录大小——超标触发 Topic Architect 单独处理 |
| 检索(2) vs 改动(7) | 检索赢 | 宁可改名后修复一批 Wikilink，也不能保留一个搜不到的标题 |
| 一棵树(3) vs 课题分离(4) | 一棵树赢 | canonical 优先级高于课题迭代——先确保不重复，再判断稳定性 |
| 规模(6) vs 改动(7) | 规模赢 | 超标信号本身就是「现在改动比以后改更便宜」的判断 |
| MOC 地图(5) vs 规模(6) | 规模赢 | MOC >40 是硬信号，此时 MOC 已经是列表而非地图 |

**最终裁决**：归属优先 > 检索优先 > 一棵树一张嘴 > 其他。前三者为结构原则，后四者为操作原则。

| 已覆盖 | 未覆盖 |
|--------|--------|
| 7 原则的判据/场景/反例/优先级 | 子目录新增的正规触发条件（何时可以建子目录？何时禁止？） |

| 与 vault-rules.yaml 阈值的对应 | 删除原则（何时可以删除一个目录/笔记？当前只有归档无删除） |
