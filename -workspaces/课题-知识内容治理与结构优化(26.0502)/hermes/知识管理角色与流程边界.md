# 知识管理角色与流程边界

**结论**：HXC Vault 知识管理涉及 7 个角色——5 核心（Material Collector / Note Crafter / Knowledge Synthesizer / Knowledge Transfer / Vault Steward）+ 1 基础设施（Knowledge Indexer）+ 1 横切决策（Topic Architect）。角色按生命周期串联，通过输入输出边界衔接，每个角色有自身的规则依赖和触发条件。人机分界清晰：定性判断归人，批量执行归机器。

## 论点 1：角色按生命周期串联——从捕获到索引形成完整链路

**论点**：5 个核心角色覆盖一条完整的知识处理流水线：裸捕获 → 格式化 → 深度编译 → 转层提炼 → 存量治理。每条都有明确的输入和输出契约，上游产出物直接成为下游输入。

**论据**——7 角色全景：

| # | 角色 | 类别 | 职责 | 当前载体 |
|---|------|------|------|----------|
| 1 | Material Collector | 核心 | 捕获原始材料，产出裸笔记 | 用户（人类主导） |
| 2 | Note Crafter | 核心 | 按规范格式化——命名/归档/frontmatter/挂 MOC | note-crafter skill |
| 3 | Knowledge Synthesizer | 核心 | 课题层深度编译——对比/辨析/形成结论 | 用户+Agent 协作（无专项 skill） |
| 4 | Knowledge Transfer | 核心 | 课题→领域转移判定+提炼执行 | 无专项 skill |
| 5 | Vault Steward | 核心 | MOC 维护/链接修复/存量审计/过期归档 | pjx-common-auditor（偏审计） |
| 6 | Knowledge Indexer | 基础 | 建索引/向量化/提供搜索 | gbrain |
| 7 | Topic Architect | 横切 | 目录结构设计/MOC 分组策略/跨域关系 | 无专项 skill |

**材料**——交互流程图：

```
用户（信息摄入）
  │
  ▼
Material Collector ──裸笔记──→ Note Crafter
                                  │
                         格式化笔记（课题/知识）
                                  │
                    ┌─────────────┤
                    ▼             ▼
              课题层笔记        领域层笔记
                    │             │
                    ▼             │
          Knowledge Synthesizer   │
                    │             │
               稳定结论？          │
               ├─ 否 → 继续迭代   │
               └─ 是              │
                    ▼             │
          Knowledge Transfer      │
              评分 ≥8？            │
              ├─ 是 → 提炼入领域层─┤
              └─ 否 → 索引链接    │
                    │             │
                    ▼             ▼
              Vault Steward（定期审计 + 修复）
                    │
                    ▼
              Knowledge Indexer（事件触发索引）
```

Topic Architect 在两个节点介入：
- Note Crafter 新建笔记时——归属哪个目录？需新建目录？
- Vault Steward 发现目录超标时——拆还是并？怎么拆？

## 论点 2：每个角色有明确的 7 维度边界定义和规则依赖

**论点**：角色定义不是描述性的一次性文档，而是可执行契约——7 维度（触发/输入/输出/产出物/质量判据/边界/规则依赖）构成角色与规则层之间的约束接口。

**论据**——各角色 7 维度详细定义：

### 1. Material Collector

| 维度 | 内容 |
|------|------|
| 触发 | 遇到有价值信息（阅读/会议/观察/突发灵感） |
| 输入 | 外部信息源（网页/论文/对话/视频/自身经验） |
| 输出 | 裸笔记——不完整的摘录、链接、一句话想法 |
| 规则依赖 | 无需结构化规则；日记模板提供最小框架 |
| 产出物 | `01-我的/日常笔记/日记-YYYY-MM-DD.md` 的 `## 捕获` 节；或临时文件 |
| 质量判据 | 未来自己能理解来源和核心信息点 |
| 边界 | 不做格式化、不归档、不判断归属目录 |

### 2. Note Crafter

| 维度 | 内容 |
|------|------|
| 触发 | 裸笔记需结构化；新建课题/知识页；日记标记为「需整理」 |
| 输入 | 裸笔记内容 + 用户意图（课题/知识页/临时记录） |
| 输出 | 符合 CLAUDE.md + vault-rules.yaml 的格式化笔记，已挂 MOC |
| 规则依赖 | CLAUDE.md / 课题目录建构规则 / vault-rules.yaml（type/moc/threshold） |
| 产出物 | `---/-workspaces/` 或 `03-领域/` 或 `06-商业/` 下的 .md 文件 |
| 质量判据 | 文件名=标题、frontmatter 完整、type 正确、已挂 MOC、无重复概念 |
| 边界 | 不做内容深加工（Synthesizer）；不做跨域目录决策（Topic Architect） |

### 3. Knowledge Synthesizer

| 维度 | 内容 |
|------|------|
| 触发 | 课题层材料足够（有 raw/entities/concepts）；用户发起「深度编译」 |
| 输入 | 课题层的 raw 材料、多来源、Stage 分析、对比记录 |
| 输出 | 稳定结论——概念定义、模型、方法、原则、对比矩阵 |
| 规则依赖 | 课题知识转入领域知识的判定标准（5 条件）；CLAUDE.md（结论段模板/对比格式） |
| 产出物 | 课题层 `concepts/` 下的编译页面；conclusions.md；comparisons.md |
| 质量判据 | 结论自带理由和出处；可脱离原始材料独立理解；覆盖反例 |
| 边界 | 不决定是否转入领域层（Transfer）；不修改领域层已有页面（Steward） |

### 4. Knowledge Transfer

| 维度 | 内容 |
|------|------|
| 触发 | Synthesizer 产出稳定结论后；Steward 发现课题层可迁移内容 |
| 输入 | Synthesizer 编译产物 + 5 条件评分 |
| 输出 | 转移决策 + 执行动作（新建/合并/索引/留课题层） |
| 规则依赖 | 课题知识转入领域知识的判定标准（5 条件 + 评分表 + 转入形态表） |
| 产出物 | 转移→领域层新建/更新 canonical 页面 + 课题层来源链接；不转移→评分记录 |
| 质量判据 | 转移后知识页可独立理解；canonical 明确；无重复解释 |
| 边界 | 不修改领域层以外内容；不覆盖已有 canonical 页面（Steward 做去重合并） |

**路径选择判断**（需机器化——当前缺口）：

```
评分 ≥8
  ├─ 已有 canonical 页 → 合并到旧页
  ├─ 无 canonical 页 → 新建主解释页
  └─ 无法脱离上下文 → 只做索引链接

评分 5-7 → 只做索引链接
评分 0-4 → 留在课题层
```

### 5. Vault Steward

| 维度 | 内容 |
|------|------|
| 触发 | 定期（cron）/ 链接断裂 / 审计报告 / 结构违规 |
| 输入 | vault-rules.yaml 规则集 + pjx-common-auditor 审计报告 + MOC 链接图 |
| 输出 | 修复的链接、清理的重复页、更新的 MOC、归档的过期内容、治理台账 |
| 规则依赖 | vault-rules.yaml（全量）；CLAUDE.md；课题目录建构规则；转入判定标准 |
| 产出物 | 治理台账 `07-治理/skill规范/`；修复 commit；审计报告 |
| 质量判据 | 断链数下降；MOC 链接数在阈值内；无 type 错误；目录 note_count 在阈值内 |
| 边界 | 不做内容创作（Crafter/Synthesizer）；迁移决策需 Topic Architect 确认 |

### 6. Knowledge Indexer（基础设施）

| 维度 | 内容 |
|------|------|
| 触发 | 新内容进入 Vault（事件触发）+ 定期全量重建 |
| 输入 | Vault 全部 .md 文件 |
| 输出 | 向量索引 + 关键词索引 + 图谱关系 |
| 载体 | gbrain (Docker Postgres + pgvector + Qwen3-Embedding-8B) |
| 边界 | 不做内容修改；不做结构化决策 |

### 7. Topic Architect（横切决策）

| 维度 | 内容 |
|------|------|
| 触发 | 新领域/子领域出现；MOC 链接数超过阈值；目录 note_count 超过阈值 |
| 输入 | vault-rules.yaml 阈值体系 + vault-index.yaml 目录结构 + MOC 链接图 |
| 输出 | 目录拆分/合并方案；命名建议；MOC 分组策略 |
| 规则依赖 | vault-rules.yaml（moc + 阈值）；课题目录建构规则（主题目录规范） |
| 产出物 | 方案文档（可执行，含影响范围评估） |
| 质量判据 | 拆分后不产生孤页；不破坏已有链接；新 MOC ≤40 链接 |
| 边界 | 只出方案，不直接执行（执行由 Note Crafter 或 Vault Steward 承担）；决策需用户确认 |

**论据**——规则依赖矩阵：

| 角色 | CLAUDE.md | vault-rules.yaml | 课题目录建构规则 | 转入判定标准 | vault-index.yaml |
|------|:---:|:---:|:---:|:---:|:---:|
| Material Collector | — | — | — | — | — |
| Note Crafter | ✓ | ✓ | ✓ | — | ✓（去重） |
| Knowledge Synthesizer | ✓ | — | — | ✓ | — |
| Knowledge Transfer | — | — | — | ✓ | ✓（canonical 查询） |
| Vault Steward | ✓ | ✓ | ✓ | ✓ | ✓（全量） |
| Knowledge Indexer | — | — | — | — | — |
| Topic Architect | — | ✓ | ✓ | — | ✓（结构洞察） |

## 论点 3：人机分界清晰——定性判断归人，批量执行归机器

**论点**：7 角色的人机分工遵循一条原则：需要上下文理解的下判断归人（Synthesizer、Topic Architect 确认），机械重复的执行归 Agent（格式化为 Crafter、修复为 Steward、索引为 Indexer）。

**论据**——人机分界表：

| 环节 | 人类 | Agent | 说明 |
|------|:---:|:---:|------|
| Material Collector | ✓ | 辅助 | 捕获灵感/判断信息价值——人类优势 |
| Note Crafter | 确认 | ✓ | Agent 可自动化，但命名/归档需确认 |
| Knowledge Synthesizer | ✓ | 辅助 | 辨析/下判断——人类主导，Agent 辅助对比整理 |
| Knowledge Transfer | 确认 | ✓ | Agent 评分+执行，转移决策需人确认 |
| Vault Steward | 确认高危 | ✓ | 批量修复可自动化，迁移/删除需确认 |
| Knowledge Indexer | — | ✓ | 全自动 |
| Topic Architect | 确认 | ✓ | Agent 出方案，人做最终决策 |

## 论点 4：当前 skill 覆盖 3/7 角色，3 个缺口需优先补

**论点**：流程层落地靠 skill。已有 3 个 skill 覆盖 3 个角色，Synthesizer / Transfer / Topic Architect 无专项 skill，是当前流程层最大短板。

**论据**——skill 映射：

| 角色 | 已有 skill | 缺口 |
|------|-----------|------|
| Note Crafter | note-crafter（Hermes + Codex 双版） | Codex 版需同步日记条件触发 |
| Knowledge Synthesizer | 无 | 需专项 skill（编译模板 + 检查清单） |
| Knowledge Transfer | 无 | 需专项 skill（评分流程 + 路径选择 + 执行） |
| Vault Steward | pjx-common-auditor | 缺 MOC 维护 + 归档功能，仅覆盖审计 |
| Knowledge Indexer | gbrain | 已有 |
| Topic Architect | 无 | 需专项 skill（拆分/合并判据 + 方案模板） |

**优先级**：规则层补缺（R6 目录规划通用原则）→ 声明层补缺（workspaces 节）→ 建 3 个缺口 skill。

**理由**：Synthesizer 和 Transfer 的行为判断依赖 R6 的判据；Topic Architect 依赖声明层的阈值体系。规则和声明不完备时建 skill 会产生返工。

- `~/.hermes/skills/note-taking/pjx-common-auditor/SKILL.md`
- gbrain skill
