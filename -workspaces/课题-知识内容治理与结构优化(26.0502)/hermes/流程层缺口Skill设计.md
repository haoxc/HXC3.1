# 流程层缺口 Skill 设计

**结论**：流程层在 7 角色中有 3 个缺口——Knowledge Synthesizer（课题层深度编译）、Knowledge Transfer（课题→领域转移）、Topic Architect（目录结构设计）。每个 skill 设计需覆盖：触发条件、输入/输出规格、规则依赖、流程步骤、成功判据。当前为设计文档，未落地为 SKILL.md。

## 论点 1：Knowledge Synthesizer——课题层深度编译

**论点**：Synthesizer 在课题层内部工作，输入来自 Note Crafter 的格式化笔记，输出稳定结论供 Transfer 评估。它的核心工作是「对比、辨析、形成结论」，不是「写更多笔记」。

**触发条件**：

| 触发信号 | 判据 |
|----------|------|
| 课题已积累 ≥3 篇格式化笔记 | Note Crafter 产出 ≥3 |
| 课题笔记多方向并行，需要收敛 | 多篇笔记讨论同一主题的不同侧面 |
| 用户指令：提炼/总结/对比某个课题 | 明确触发 |

**输入**：课题目录下 ≥2 篇相关笔记（文件路径列表 + 主题描述）
**输出**：一篇 synthesis 笔记（包含对比表格、结论判断、未解决争议清单）

**规则依赖**：R1（CLAUDE.md 笔记风格）+ R3（转入领域判定标准，用于前置检查）

**流程步骤**：

```
1. 扫描课题目录 → 提取相关笔记列表
2. 识别共同主题与冲突点
   ├─ 同一概念的多个侧面 → 提取最大公约属性
   ├─ 同一问题的多个判断 → 判差异是否可调和
   └─ 同一现象的多个解释 → 分层级标定（原则层 / 场景层 / 案例层）
3. 写 synthesis 笔记
   ├─ 结论段：最终判断是什么（三段式：结论/理由/出处）
   ├─ 对比表：不同判断的差异维度（维度 | 方案A | 方案B）
   ├─ 未解决争议：哪些分歧尚未收敛，各自前提条件
   └─ 出处清单：溯源自哪些原始笔记
4. 输出 synthesis + 标记原始笔记 frontmatter `synthesized_in: [[synthesis页]]`
```

**反例——不应该做的**：
- ✗ 只罗列不判断——"A 说 X，B 说 Y"不是 synthesis，必须给出「选 X 或 Y 或 X∧Y 的理由」
- ✗ 替代 Note Crafter 做格式化——synthesis 输入已经是格式化笔记，不应该从头写
- ✗ 在 synthesis 中引入新材料——只处理已有笔记，不新增外部材料（新增应回到 Note Crafter 流程）

**成果物判据**：

| 判据 | ✓ | ✗ |
|------|---|---|
| 每个结论三段式 | `**结论**：选 Redis。\n**理由**：XX\n**出处**：XX` | 段落式判断无出处 |
| 冲突已标定 | 未解决争议清单含前提条件和分歧点 | 模糊收场：「各有优劣，根据情况选择」 |
| synthesis <80 行 | 结论在 15 行内可扫描 | synthesis 变成另一篇长文 |
| 原始笔记已标记 | frontmatter 含 `synthesized_in` | 原始笔记无任何连接提示 |

## 论点 2：Knowledge Transfer——课题→领域转移判定与执行

**论点**：Transfer 是课题层与领域层的闸门。它执行 R3 的 5 条件评分，决定是否提炼、以什么形态（新建/合并/索引）转入领域层，并执行提炼操作或标记 `needs_transfer`。

**触发条件**：

| 触发信号 | 判据 |
|----------|------|
| Synthesis 笔记已完成 | Synthesizer 输出 synthesis |
| 课题标记 `completed` | 用户标记 or Steward 审计发现 |
| Vault Steward 发现 `needs_transfer` 未评分 | cron 审计触发 |
| 用户指令：评估某课题转移 | 明确触发 |

**输入**：synthesis 笔记 + 课题目录上下文
**输出**：评分结果 + 提炼后的领域层 canonical 页（或 `needs_transfer` 标记 + 待办）

**规则依赖**：R3（转入领域判定标准）+ R4（canonical 页面清单）+ vault-rules.yaml threshold

**流程步骤**：

```
1. 读 synthesis 笔记 → 提取可转移的结论单元
2. 每个结论单元执行 5 条件评分
   ├─ 稳定性（0-2）/ 泛化性（0-2）/ 可复用性（0-2）
   ├─ 证据充分性（0-2）/ 归口清晰（0-2）
   └─ 总分
3. 判定：
   ├─ 0-4 分 → 留在课题层，标记 `transfer_score: 0-4`
   ├─ 5-7 分 → 在领域层 MOC 做索引链接，标记 `transfer_score: 5-7`
   └─ 8-10 分 → 执行提炼转入领域层
4. 提炼执行：
   ├─ 检查 canonical 页面 → 是否已有主解释页？
   │   ├─ 有 → 更新旧页（追加结论/修正过时判断）
   │   └─ 无 → 创建新 canonical 页
   ├─ 检查命名（R1+R2）→ 文件名=核心主题词，title 含结论
   ├─ 挂 MOC → 确保领域 MOC 链接数未超 40
   └─ 课题层保留来源链接：正文底部「形成来源：[[课题入口]]」
5. 更新课题 frontmatter：
   ├─ status: merged_to_domain
   └─ deliverables: [[领域层页面1]], [[领域层页面2]]
```

**提炼形态决策**：

| 条件 | 操作 |
|------|------|
| 新概念、无 canonical 页 | 创建新 canonical 页 |
| 已有 canonical 页，结论一致 | 更新旧页（追加证据/案例） |
| 已有 canonical 页，结论冲突 | 标记冲突 → 生成对比笔记 → 用户裁决 |
| 结论依赖课题上下文 | 领域 MOC 加索引链接，不提炼正文 |

**反例——不应该做的**：
- ✗ 批量迁移——一次 Transfer 只处理一个 synthesis，不批量操作多个课题
- ✗ 复制正文——领域层只提炼结论，不复制课题过程

**成果物判据**：

| 判据 | ✓ | ✗ |
|------|---|---|
| 评分可追溯 | 每个结论单元有 5 维度评分明细 | 只有总分无明细 |
| canonical 页可独立理解 | 脱离课题上下文可读 | 引用课题层材料才能理解 |
| 来源可追溯 | 正文底部有「形成来源」链接 | 无来源链接 |
| 课题不残留副本 | 课题层结论改为 `[[链接]]` 引用 | 课题层和领域层各一份相同正文 |

## 论点 3：Topic Architect——目录结构设计与 MOC 分组

**论点**：Topic Architect 是唯一横切角色——既读声明层阈值受约束，又为 Note Crafter（新建时判断归属）和 Vault Steward（存量超标判断拆分）出结构方案。它「只出方案不做执行」，执行由 Note Crafter 或 Steward 承担。

**横切图**：

```
          读声明层阈值（moc/scale_thresholds）
                     │
        ┌────────────┼────────────┐
        ▼            │            ▼
  Note Crafter       │     Vault Steward
  新建时归属不确定   │     存量超标触发
        │            │            │
        └────────────┼────────────┘
                     ▼
            Topic Architect 出方案
              ├─ 新增目录？拆分子目录？合并目录？
              ├─ MOC 分组调整？
              └─ 产出：方案文档 + 影响评估
                     │
               用户确认？
               ├─ 否 → 修订方案
               └─ 是
                     ▼
            Note Crafter / Steward 执行
```

**触发条件**：

| 触发信号 | 判据 |
|----------|------|
| Note Crafter 新建笔记时归属模糊 | 主题横跨多个顶层目录，无法归属判断（原则 1） |
| Vault Steward 审计发现缺口 | 目录 >500 note / MOC >40 链接 / 孤立笔记 >0 |
| 用户指令：评估某目录结构 | 明确触发 |

**输入**：触发上下文（哪个目录、什么问题）+ vault-index.yaml（当前目录全貌）
**输出**：结构方案文档（目标结构 + 迁移影响 + 风险 + 步骤）

**规则依赖**：R2（课题目录建构规则）+ R5（7角色定义）+ vault-rules.yaml threshold/moc

**流程步骤**：

```
1. 确认问题 —— 归属模糊？规模超标？MOC 退化？孤立堆积？
2. 读 vault-index.yaml → 构建当前目录心智模型
3. 应用 R6 7原则 + threshold：
   ├─ 归属优先 → 跨域内容定主要归属
   ├─ 规模敏感 → 超标时评估拆分 vs 合并
   ├─ 改动最小 → 先评估修补成本 vs 搬迁成本
   └─ MOC 是地图 → 分组策略（概念层级/工作流/对比/时间线）
4. 出方案：
   ├─ 方案 A（保守：修补入口，不搬迁）→ 成本低，效果有限
   ├─ 方案 B（激进：搬迁重组，全量修复）→ 成本高，根治
   └─ 推荐方案 + 理由
5. 方案内容：
   ├─ 目标目录结构（Mermaid/ASCII tree）
   ├─ 迁移清单（哪些文件移动，哪些保持不变）
   ├─ 受影响链接清单（哪些 Wikilink 需修复）
   ├─ 风险清单（断链/重复/合并冲突）
   └─ 执行步骤（分阶段，每阶段验证点）
```

**目录拆分/合并决策表**：

| 信号 | 偏拆分 | 偏不拆/合并 |
|------|--------|------------|
| note_count >500 | 是 | 若内聚度高 → 先分子 MOC 不分子目录 |
| MOC links >40 | 是 | 若分组清晰 → 不拆 |
| 孤立笔记 >0 | 是（新建子目录吸纳） | 若孤立笔记稳定且不应挂入当前域 → 归入其他域 |
| 子主题已稳定且≥3篇 | 是 | — |
| 子主题仍在快速演化 | — | 推迟决策，标记监控 |
| 拆后跨子目录引用成本增加 | — | 评估是否引用成本 > 拆分收益 |

**反例——不应该做的**：
- ✗ 只出方案不评估影响——方案必须含受影响链接清单和风险
- ✗ 为拆分而拆分——52 个笔记的超标目录如果内聚度高、搜索方便，可以不拆
- ✗ 越权执行——Topic Architect 不应该直接修改文件（那是 Note Crafter/Steward 的工作）

**成果物判据**：

| 判据 | ✓ | ✗ |
|------|---|---|
| 方案可执行 | 步骤按阶段分，有验证点 | 只有目标无路径 |
| 影响可评估 | 迁移清单 + 受影响链接清单 | 只说「大量文件迁移」 |
| 风险已识别 | 每项迁移潜在风险 + 缓解措施 | 无风险分析 |
| 备选方案 | 至少两方案对比 + 推荐理由 | 单方案无对比 |

## 三个 Skill 的交互时序

```
Material Collector → Note Crafter → Knowledge Synthesizer → Knowledge Transfer
                       │                                         │
                       │    (归属模糊时)                          │
                       └────────→ Topic Architect ←──────────────┘
                                        │             (拆分决策时)
                                        └────────→ Vault Steward
```

**Synthesizer 和 Transfer 之间的耦合**：
- Synthesizer 输出 synthesis 笔记，Transfer 读取；没有 synthesis 时 Transfer 不应被触发
- Synthesizer 可以在编译过程中执行 R3 的预评分（给 Transfer 提供参考），但最终评分由 Transfer 确认

**Topic Architect 的异步性**：
- Topic Architect 方案产出后，需要用户确认才能进入执行——这个等待是全异步的
- 在等待确认期间，Note Crafter 和 Steward 继续按现有规则工作，不阻塞

## 落地优先级

| 顺序 | Skill | 理由 |
|------|-------|------|
| 1 | Knowledge Synthesizer | 压缩信噪比最急需——课题层堆积未编译导致检索效率下降 |
| 2 | Knowledge Transfer | 依赖 Synthesizer 输出，且 R3 已有完整评分标准可直接编程 |
| 3 | Topic Architect | 低频触发（阈值超标才启动），可手工替代 |

**落地形式**：
- Synthesizer + Transfer → Hermes Agent skill（目录规划/整理类，agent 直接使用）
- Topic Architect → 方案生成 skill，需人工确认步骤
