# 三层管理框架设计

**结论**：HXC Vault 知识管理采用三层框架——声明层（vault-rules.yaml / 机器可校验）→ 规则层（规则文档 / 含判据+反例）→ 流程层（7 角色+skill / 执行）。层间通过规则注入衔接，Topic Architect 横切于声明层与流程层之间做结构决策。框架不是新建，是已有材料的串连与补缺。

## 论点 1：声明层承载机器可校验的约束

**论点**：所有可形式化校验的规则应进入声明层（vault-rules.yaml），余下含判据+反例的人类判断保留在规则层。声明层不应膨胀为规则库——机器只能校验结构，不能校验语义。

**论据**：

| 已覆盖 | 规则 | 校验方式 |
|--------|------|----------|
| top_level | 顶层目录角色定义（`99-设置/` 不放普通笔记） | schema 校验 |
| type | note/moc/log 的 frontmatter 字段要求 | schema 校验 |
| moc | MOC 链接数上限、正文长度上限 | 脚本扫描 |
| threshold | 笔记 >80 行警告、MOC >40 链接警告、目录 >500 note 警告 | 脚本扫描 |
| vault-settings | 附件管理 / frontmatter 字段 / 工作区隔离 | schema 校验 |

| 缺 | 说明 |
|----|------|
| workspaces 节 | `---/-workspaces/` 的 pattern/date_policy/organization_modes/domain_reference 尚未声明化 |
| 子目录拆分阈值 | 何时必须建子目录、何时禁止拆——当前只有 >500 一条，缺反例 |
| type 交叉约束 | 例如「moc 下不能直接放 note，只能放子 moc 或 wikilink」——未编码 |

**材料**：
- `99-设置/vault-rules.yaml`（v1.0.0，266 行）
- `99-设置/vault-index.yaml`（目录结构索引，3032 行）

## 论点 2：规则层承载人类可读的判据+反例

**论点**：规则层是声明层与流程层之间的翻译层。声明层说「MOC >40 链接警告」，规则层说「为什么 40、超过后拆还是并、什么情况下不拆（反例）」。规则层文档每条规则必须有判据和反例对照。

**论据**——已有规则集：

| # | 规则文档 | 覆盖范围 | 缺口 |
|---|----------|----------|------|
| R1 | CLAUDE.md | 笔记风格：标题/句式/长度/链接/语气/对比/结论段模板 | 无 |
| R2 | 课题目录建构规则 | 课题创建：命名/目录/组织模式/领域引用边界 | 无 |
| R3 | 课题知识转入领域知识的判定标准 | 转层判定：5 条件+评分+转入形态 | 路径选择判断未机器化 |
| R4 | 管理学科高频概念 canonical 页面清单 | 管理学科 canonical 页面指定 | 仅覆盖管理学科 |
| R5 | 知识管理角色与流程边界 | 7 角色定义+交互+规则依赖 | 无 |

| 缺口规则集 | 说明 |
|-------------|------|
| R6: 目录规划通用原则 | 跨领域适用的「这个内容该放哪」决策模型——7 原则已提取，尚未展开为判据/场景/反例/优先级 |

**规则层与声明层的边界契约**：

| 进声明层 | 留规则层 |
|----------|----------|
| 可机器扫描的量化阈值 | 含人类判断的定性标准 |
| 目录/type/frontmatter 结构约束 | 写作风格/语气/链接习惯 |
| 能被 script 校验的字段 | 需要上下文理解的反例 |
| 跨规则文档统一的格式 | 单个领域的特殊约定 |

**材料**：
- `99-设置/vault-rules.yaml`
- CLAUDE.md（笔记风格规则）
- `课题目录建构规则.md`
- `课题知识转入领域知识的判定标准.md`
- `管理学科高频概念canonical页面清单.md`
- `知识管理角色与流程边界.md`

## 论点 3：流程层由 7 角色串联执行，规则通过注入路径到达角色

**论点**：流程层不是「一个角色加载所有规则」，而是每个角色按职责加载所需的规则子集。规则注入的路径是：声明层 → 规则层 → 角色定义（规则依赖列表）→ skill 实现时加载对应规则。

**论据**——7 角色在三层中的锚定：

```
声明层（vault-rules.yaml）
  │
  │  结构约束/阈值/type定义
  ▼
规则层（6 组规则文档）
  │
  │  ┌───────────────────── 规则注入 ─────────────────────┐
  │  ▼                      ▼              ▼              ▼
流程层
  │
  ├─ Material Collector  [无规则依赖，人类主导]
  │       │
  │       ▼ 裸笔记
  ├─ Note Crafter  [R1+R2+声明层 type/moc]
  │       │
  │       ▼ 格式化笔记
  │   课题层笔记 ──→ Knowledge Synthesizer  [R1+R3]
  │       │                    │
  │       │              稳定结论？
  │       │               ├─ 否 → 继续迭代
  │       │               └─ 是
  │       │                    ▼
  │       └──────────→ Knowledge Transfer  [R3+R4+声明层 threshold]
  │                               │
  │                         评分 ≥8？
  │                         ├─ 是 → 提炼入领域层
  │                         └─ 否 → 索引链接
  │                               │
  │                               ▼
  └─ Vault Steward  [R1+R2+R3+R4+R5+声明层全量]
          │
          ▼
    Knowledge Indexer  [无规则依赖，基础设施]

Topic Architect  [R2+R5+声明层 threshold+moc]
  ↑                               ↑
  │ 目录决策                       │ 拆分决策
  Note Crafter                  Vault Steward
```

**论据**——规则注入矩阵（已在角色定义中建立）：

| 角色 | R1 | R2 | R3 | R4 | R5 | 声明层 |
|------|:--:|:--:|:--:|:--:|:--:|:--:|
| Material Collector | | | | | | |
| Note Crafter | ✓ | ✓ | | | | type/moc |
| Knowledge Synthesizer | ✓ | | ✓ | | | |
| Knowledge Transfer | | | ✓ | ✓ | | threshold |
| Vault Steward | ✓ | ✓ | ✓ | ✓ | ✓ | 全量 |
| Knowledge Indexer | | | | | | |
| Topic Architect | | ✓ | | | ✓ | threshold/moc |

**材料**：
- `知识管理角色与流程边界.md`（角色定义+规则依赖矩阵）
- note-crafter skill（`~/.hermes/skills/note-taking/note-crafter/`）
- pjx-common-auditor skill

## 论点 4：Topic Architect 是唯一横切角色——既读声明层又影响流程层

**论点**：Topic Architect 的特殊性在于：它不是生命周期中的一个环节，而是在 Note Crafter（新建时判断目录归属）和 Vault Steward（存量超标时判断拆分）两个节点做结构决策。它读声明层的阈值约束，产出规则层可引用的方案模板，最终由流程层的 Note Crafter 或 Steward 执行。

**论据**：

```
Topic Architect 横切路径：
                    读声明层阈值
                         │
            ┌────────────┼────────────┐
            ▼            │            ▼
      Note Crafter       │     Vault Steward
      新建时归属判断      │     存量超标拆分判断
            │            │            │
            └────────────┼────────────┘
                         ▼
                  Topic Architect 出方案
                  （目录结构变更 / MOC 分组调整）
                         │
                   用户确认？
                   ├─ 否 → 修订方案
                   └─ 是
                         ▼
                Note Crafter 或 Steward 执行
```

**Topic Architect ≠ 目录管理员**。它只出方案不做执行——执行由 Note Crafter（新建目录+笔记）或 Vault Steward（迁移+修复链接）承担。这个边界防止「设计者自己执行导致无审查」。

**材料**：
- vault-rules.yaml threshold 节
- 课题目录建构规则（推荐主题目录）
- vault-index.yaml（当前目录结构全貌）

## 当前框架完整性评估

| 层 | 完备度 | 缺口 |
|----|--------|------|
| 声明层 | 70% | workspaces 节、子目录拆分反例、type 交叉约束 |
| 规则层 | 75% | R6（目录规划通用原则）未展开；R3 路径选择未机器化 |
| 流程层 | 55% | 3 个 skill 缺口（Synthesizer/Transfer/Topic Architect）；Steward 缺 MOC 维护功能 |

**优先级**：规则层补缺（R6+R3）→ 声明层补缺（workspaces 节）→ 流程层建 skill（3 缺口）。

**理由**：规则不完整时建 skill 会产生返工——skill 的行为判断依赖规则层的判据和反例。先定规则，再写 skill。
