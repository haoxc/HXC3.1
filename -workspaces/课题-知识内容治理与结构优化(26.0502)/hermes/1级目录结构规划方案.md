# 1 级目录结构规划方案

**结论**：Vault 当前 15 个 L1 目录存在 6 个结构性问题——`Clippings` 未声明、`01-我的` 严重超标（1,700+ 文件）、`04-笔记` 空置、`06-商业` 过于稀疏、`z附件` L1 地位不明、`---` 下混入 TEST/术语库等非课题内容。规划方案以归属优先和改动最小为原则，建议分阶段处理：先补声明层（将 Clippings 纳入框架）、再理归属（`01-我的` 下项目内容分流）、后去噪音（清理 TEST/空目录/附件）。

## 问题 1：`Clippings` 未纳入声明层——外部约束（已澄清）

**状态**：57 个 .md 文件（52 个在 `z附件/` 下，4 个散落 .md + 1 个 MOC `Clippings.md`）。**L1 位置不可变**——Obsidian Clipper 插件强制写入 Vault 根目录，用户无法修改目的地。

**约束分析**：

| 维度  | 事实                                 |
| --- | ---------------------------------- |
| 源   | Obsidian Clipper 浏览器插件             |
| 路径  | 固定在 Vault 根 `/Clippings/`          |
| 内容  | 网页摘录 → `.md`；图片嵌入 → `z附件/*_MD5.md` |
| 可控性 | 路径不可改；`z附件/` 自动生成不可阻止              |
| 性质  | 输入通道（input channel），非知识组织目录        |

**定位**：`Clippings` 与 `02-输入` 是**平行输入通道**——`02-输入` 是人工捕获通道（用户主动放入），`Clippings` 是自动化捕获通道（插件写入）。两者不冲突，也不应合并。

**方案**：

1. **纳入声明层**：在 `vault-rules.yaml` 声明 Clippings，role = `input_channel`，`managed: false`（插件管辖，不施加治理规则）
2. **生命周期**：剪藏落地 → 用户审阅 → 处理后移入归属目录（`03-领域/` 或 `01-我的/`）→ 原始剪藏可删除或归档
3. **`z附件/` 处理**：插件自动生成的 MD5 图片引用页，纳入 `.gitignore`（已有规则的扩展）

### vault-rules.yaml 声明提案

```yaml
Clippings:
  path: Clippings
  role: input_channel
  description: Obsidian Clipper 插件自动写入，路径不可修改
  managed: false
  required: false
  core: false
  children:
    - name: z附件
      role: attachments_local
      note_count: 52
      description: Clipper 插件自动生成的图片嵌入引用页
```

**优先级**：P1——外部约束已明确，只需声明层补缺即可（零文件移动，1 行 YAML）。

**改动成本**：低——不移动文件，仅补 vault-rules.yaml 声明。

## 问题 2：`01-我的` 严重超标（归属边界模糊）

**状态**：L1 层自身仅 1 个 .md；但递归子目录中 13 个 L2 子目录，总 .md 数 ~1,700+。

**规模分布**：

| 子目录 | .md 数 | 性质推断 | 建议归属 |
|--------|:---:|------|------|
| `常工院/` | 706 | 教育项目（课程/作业/笔记） | → `---/-workspaces/` 或 `03-领域/` |
| `项目/` | 568 | 项目过程材料 | → `---/-workspaces/` |
| `工作室/` | 121 | 工作室项目 | → `---/-workspaces/` |
| `日常笔记/` | 91 | 日记/思考/观察 | 保留 `01-我的/` |
| `学习/` | 85 | 学习笔记 | → `03-领域/` 或保留 `01-我的/` |
| `个人/` | 58 | 个人记录 | 保留 `01-我的/` |
| `创新/` | 42 | 创意/想法 | 保留 `01-我的/` |
| `课题/` | 38 | 课题过程 | → `---/-workspaces/` |
| `草稿/` | 17 | 待处理碎片 | 保留 or → `04-笔记/` |
| `学术/` | 5 | 学术笔记 | → `03-领域/` |
| `决策/` | 4 | 个人决策 | 保留 `01-我的/` |
| `备忘/` | 2 | 备忘 | 保留 `01-我的/` |
| `提示词历史库/` | 1 | 工具历史 | → `09-工具/` or 保留 |

**核心问题**：`01-我的` 的 role 是 "personal"——个人信息、目标、日记、思考。但 `常工院`（706）、`项目`（568）、`工作室`（121）合计 1,395 个文件明显不是「个人信息」，而是教育/职业项目的过程材料。

**方案**——三段分流：

| 段   | 内容类型    | 目标                          |   规模   |
| --- | ------- | --------------------------- | :----: |
| A   | 课题/项目过程 | → `---/-workspaces/` 新建主题目录 | ~1,395 |
| B   | 通用知识    | → `03-领域/` 对应子域             |  ~90   |
| C   | 纯个人     | 保留 `01-我的/`                 |  ~200  |

**原则应用**：

| #      | 原则                                      | 判断  |
| ------ | --------------------------------------- | --- |
| 1 归属优先 | 项目过程 ≠ 个人信息。归属错误是一级问题，必须修正              |     |
| 6 规模敏感 | 1,700+ >> 500 阈值，超标信号是硬信号               |     |
| 7 改动最小 | 迁移分批进行：先补声明层（不移动文件）→ 新内容走新规则 → 存量按优先级迁移 |     |

**风险**：

| 风险                        | 缓解                               |
| ------------------------- | -------------------------------- |
| 迁移 1,395 文件破坏 Wikilink    | 分阶段迁移，每阶段修复受影响链接                 |
| 用户认知惯性——习惯在 `01-我的/` 下找项目 | 在 `01-我的/` 保留 Wikilink 索引，定向到新位置 |
| 部分内容归属模糊（项目笔记含知识性内容）      | 不做全自动迁移，每阶段人工判定边界                |

**优先级**：P2——规模超标不立即产生功能性错误，但持续堆积会增加日后迁移成本。与 P1 可并行。

## 问题 3：`04-笔记` 空置（inbox 机制未启用）

**状态**：0 个 .md。`vault-rules.yaml` 设定了 `hygiene: max_pending: 50`，但目录完全空置。

**诊断**：inbox 空置可能有三种解释：
- A）不需要 inbox——捕获直接进入归属目录，短路径优于两段式
- B）inbox 在其他位置——捕获放在 `01-我的/草稿/` 或日记 `## 捕获` 节
- C）inbox 应该保留作为「概念性收件箱」——不强制使用，但声明存在

**方案**：
- 保留目录和声明层约束（max_pending: 50），不改变使用习惯。空置不产生结构风险。
- 在 vault-rules.yaml 加注释：`04-笔记` 为概念收件箱；实际捕获发生在日记 `## 捕获` 节或其他临时位置。

**优先级**：P4——无功能影响，声明层澄清即可。

## 问题 4：`06-商业` 过于稀疏

**状态**：1 个 .md。role=business，MOC 已建（`06-商业/06-商业.md`）。

**诊断**：
- 体量过小但 role 合理——商业情报是独立的知识域，与 03-领域（学术/技术）有明确的归属边界
- 1 个文件不能说明目录无价值——可能是新建立的域，等待填充

**方案**：保留。不做合并或删除。在 vault-rules.yaml 中标记 `required: false`（已是 false）并加 `min_note_warn: 5` 作为审计提示。

**优先级**：P4——低频使用目录，不产生结构风险。

## 问题 5：`z附件` L1 地位

**状态**：30 个 .md。role=attachments_local，required=false。

**与 `98-附件` 的关系**：

| | `98-附件` | `z附件` |
|---|-----------|---------|
| role | attachments_central | attachments_local |
| git | 被 .gitignore 排除 | 未声明 |
| 位置 | L1 集中 | L1 分散 |
| 规模 | 1 | 30 |

**问题**：L1 出现 `z附件` 作为「分散附件」与 `98-附件`（集中附件）的区分是合理的，但 L1 目录应该承担语义角色，附件是辅助内容。`z附件` 在 L1 上破坏了「L1 = 知识域」的语义一致性。

**方案**：三种选择——

| 方案 | 描述 | 优点 | 缺点 |
|------|------|------|------|
| A: 保留 L1 | 维持现状 | 零成本 | L1 语义不一致；30 个 .md 多为图片引用页，实际附件在 `98-附件/` |
| B: 降为子目录 | `98-附件/z附件-local/` | L1 干净 | 所有引用路径需更新 |
| C: 按域分散 | 各域的 `z附件/` 归入域内（如 `03-领域/60-艺术学科/z附件/`） | 附件与内容靠近 | 违反 vault-rules.yaml 的 `attachment_local` 统一命名规则；`.gitignore` 需按域配置 |

**推荐**：C（用户决议）——降级为 L2，归入 `99-设置/z附件/`。附件是基础设施，归属 99-设置（系统配置层）更合理。`98-附件/` 保持独立为大文件处理目的地，两者角色不混。

**优先级**：P3——影响 L1 语义整洁度但不阻塞功能。

## 问题 6：`---` 下混入非课题内容

**状态**：`---` role=workspace，但包含：
- `TEST/`（3 个 .md）——测试/沙盒内容，非课题
- `术语库(Terms svy)/`（6 个 .md）——术语管理，可能是独立域或工具
- `主题-世界模型场景/`（1 个 .md）——只有 1 个 .md，可能是课题起点

**归属判断**：

| 子目录 | 应归属 | 理由 |
|--------|--------|------|
| `TEST/` | `10-归档/TEST/` 或直接删除 | 测试数据，不应留在 workspace 域 |
| `术语库(Terms svy)/` | 待定 | 如果是知识型术语表 → `03-领域/`；如果是工具参考 → `09-工具/`；如果是 Vault 治理词汇 → `07-治理/` |
| `主题-世界模型场景/` | `---/-workspaces/AI课题/` or 独立课题 | 只有 1 .md + 空子目录，可能是未启动的课题 |

**方案**：
1. `TEST/` → 评估后归档或删除
2. `术语库(Terms svy)/` → 用户确认归属后迁移
3. `主题-世界模型场景/` → 如果继续推进，移到 `---/-workspaces/AI课题/` 下建课题目录

**优先级**：P2——workspace 域混入非课题内容违反原则 1（归属优先），且 TEST 目录是噪音。

## 整体规划路线

```
Phase 1: 声明层补缺（低风险，无文件移动）
  ├─ 在 vault-rules.yaml 声明 Clippings：role=input_channel, managed=false
  ├─ 确认 z附件 处理方案
  ├─ 为 04-笔记 加注释说明空置原因
  └─ 为 06-商业 加 min_note_warn

Phase 2: 内容归属纠正（中风险，含文件移动）
  ├─ ---/TEST → 归档/删除
  ├─ ---/术语库 → 确认归属后迁移
  ├─ 01-我的/项目/ + 工作室/ + 常工院/ → 分流评估
  │   ├─ 段 A：课题过程 → ---/-workspaces/
  │   ├─ 段 B：通用知识 → 03-领域/
  │   └─ 段 C：纯个人 → 保留
  └─ Clippings → 归属分流（02-输入/ or 10-归档/）

Phase 3: 结构去噪音（低风险，目录层面操作）
  ├─ z附件 降级为 L2，归入 99-设置/
  ├─ 清理空目录（---/-workspaces 下 note_count=0 的子目录）
  └─ 01-我的/ 分流后重新评估规模是否 <500
```
