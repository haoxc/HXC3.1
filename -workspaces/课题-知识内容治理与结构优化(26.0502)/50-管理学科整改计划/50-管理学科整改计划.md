# 50-管理学科整改计划

**结论**：`50-管理学科` 的整改目标不是“把目录改得更整齐”，而是把它从材料堆积区整理成可检索、可复用、可复盘的管理学科知识域。整改顺序应是：先修入口和规则，再定概念归口，最后再做目录迁移。

## 整改目标

| 目标       | 到位状态                                               |
| -------- | -------------------------------------------------- |
| 根入口可用    | `[[50-管理学科]]` 是真正的领域 MOC，能进入术语、辨析、方法工具、知识体系、研究课题索引 |
| 概念只在一处详解 | `经营维度`、`战略维度`、`指标设计` 等有 canonical 页面               |
| 课题和知识分层  | 新课题进入 `---/-workspaces/管理课题/`，`03-领域/` 只沉淀稳定结论     |
| MOC 不再竞争 | `--/MOC` 不作为活跃导航系统，价值内容合并到根 MOC 或子 MOC             |
| 链接可信     | 核心 MOC 和课题入口无断链，歧义链接有明确目标                          |
|          |                                                    |

## 目标结构

```text
03-领域/50-管理学科/
├── 50-管理学科.md                  # 领域 MOC
├── --管理术语表/                   # 管理专用术语
├── 01-管理-辨析与摘要/              # 概念辨析、速查、摘要
├── 02-管理-课题索引/                # 仅保留课题入口和状态，不保存深度过程
├── 03-管理-方法工具/                # FMEA、OKR、BSC、鱼骨图等
└── 04-管理-知识/                   # 战略规划、经营管理、绩效管理等稳定知识
```

对应 workspace：

```text
---/-workspaces/管理课题/
├── 课题-战略指标量化/
├── 课题-持续改善/
└── 课题-指标设计/
```

说明：

- `02-管理-课题索引/` 可以先不立即改名，第一阶段只调整其角色。
- 深度课题应迁入 `---/-workspaces/管理课题/`，领域层只保留课题入口和沉淀结论。
- 旧课题不批量迁移；只有继续推进、需要整改或需要复盘时再迁移。
- 领域层默认不反向引用课题过程内容，必要时只链接课题入口。
- 若某目录暂不迁移，先在 MOC 中标注“待归口”，不要强行移动。

## 阶段计划

### P0 基线冻结

目的：先知道现状，不在混乱状态下搬文件。

动作：

- 记录当前目录树、Markdown 数量、核心入口文件。
- 建立待整改清单：断链、重复概念、候选迁移课题、候选 canonical 页面。
- 不改动内容，只形成基线。

交付物：

- `50-管理学科-整改基线.md`
- `50-管理学科-链接问题清单.md`

### P1 重建根 MOC

目的：让 `[[50-管理学科]]` 成为可靠入口。

动作：

- 将 `type: note` 改为 `type: moc`。
- tags 从 `[工具]` 调整为 `[管理学科, MOC]` 或等价规范标签。
- 建立五组导航：术语、辨析与摘要、方法工具、知识体系、研究课题。
- 只放链接和极短说明，不放正文。

验收：

- 根 MOC 能进入所有一级子域。
- 根 MOC 无长段落、无过程记录、无决策正文。

### P2 概念归口

目的：解决“一棵树多张嘴”。

动作：

- 为高频概念指定 canonical 页面。
- 术语页只写定义和边界，知识页写模型、方法、应用。
- 重复解释页改为引用、辨析或案例，不再重复完整定义。

优先概念：

- `经营维度`
- `战略维度`
- `指标设计`
- `目标(Goal) / 目标(Target)`
- `指标 / 指示器 / 指数 / KPI`

交付物：

- [[管理学科高频概念canonical页面清单]]

验收：

- 每个重点概念有唯一主解释页。
- 其他页面通过 [[wikilink]] 指向主解释页。

### P3 课题分层

目的：把“研究过程”和“稳定知识”分开。

动作：

- 将 `02-管理-课题` 中内容分三类：
  - 已沉淀知识：回流到 `04-管理-知识/`。
  - 深度研究课题：迁入 `---/-workspaces/管理课题/`。
  - 轻量课题入口：保留在 `02-管理-课题索引/`。
- 对 `课题-指标设计` 做专项判断：如果仍是 processing，优先迁入 workspace；如果已稳定，拆为知识页。
- 新建课题默认不加日期；创建前确认是否短期课题、长期课题或阶段性调研。

验收：

- `03-领域/` 不保存研究全过程。
- 每个深度课题有独立入口、过程材料和回流目标。

### P4 方法工具归类

目的：降低“工具”一词和 Vault 顶层 `09-工具/` 的歧义。

动作：

- 将 `03-管理-工具` 的角色明确为“管理方法工具”。
- 可选重命名为 `03-管理-方法工具`。
- 每个工具页补足：定义、适用场景、输入输出、误用风险、相关课题。

验收：

- FMEA、OKR、BSC、鱼骨图等不再被误解为软件工具。
- 方法工具能从根 MOC 和相关课题双向发现。

### P5 清理 `--/MOC`

目的：取消第二套导航。

动作：

- 评估 `--/MOC/MOC-运营管理`、`MOC-目标如何规划和落地` 的有效链接。
- 有价值链接合并到根 MOC 或对应知识页。
- 原 MOC 页面改为归档说明或移动到待归档区。

验收：

- 活跃导航只剩根 MOC 与必要子 MOC。
- `--/MOC` 不再承担入口职责。

### P6 链接与元数据收尾

目的：让结构整改可验证。

动作：

- 修复核心 MOC、课题入口、术语入口中的断链。
- 统一核心页面 frontmatter：`tags`、`description`、`type`、`create-date`。
- 对迁移后的页面做抽样回链检查。

验收：

- 核心入口无断链。
- 重点页面都有至少一个发现路径：MOC、反链或有效标签。

## 迁移决策表

## 风险控制

- 不做一次性大搬家；每个阶段单独提交。
- 任何重命名前先列出源路径、目标路径和影响链接。
- 先修 MOC，再迁移目录；没有入口的迁移不可执行。
- 对 `课题-*` 不直接搬入知识层，先用转层评分判断。
- 不删除历史页面；先改为归档或重定向。
- 领域层不直接引用课题过程内容；确需追溯时，只链接课题入口。
