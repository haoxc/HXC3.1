# 术语解析：预演执行 (Dry Run)

> [!IMPORTANT] 快速理解
> Dry Run 是“只预演、不落地”的执行方式：系统尽量按真实流程计算将要发生的变化，但不真正写入、删除、发布、发送或提交。

## 1. 概念界定 (Concept Definition)

预演执行 (Dry Run) 指在执行命令、脚本、部署、迁移或批量操作前，以无副作用方式模拟执行路径，并输出预计影响。

它的核心不是“测试是否成功”，而是“提前看清如果真的执行，会影响哪些对象、产生哪些变更”。

常见输出包括：将新增、修改、删除的对象，将执行的 SQL、API、部署步骤或迁移步骤，参数、权限、路径、筛选条件是否可解析，受影响对象数量，以及可能阻塞真实执行的错误。

## 2. 替代范式 (Alternative Paradigms)

- 预演执行 (Dry Run)：不产生实际副作用，重点是预览影响。
- 直接执行 (Run)：立即修改真实对象，风险最高。
- 计划预览 (Plan / Preview)：只生成变更计划，可能不完整模拟真实执行路径。
- 测试运行 (Test Run)：真实运行在测试环境中，可能产生测试环境副作用。
- 回滚 (Rollback)：变更后撤销，不是执行前预防。

## 3. 边界辨析 (Boundary Analysis)

| 维度 | Dry Run | Test Run | Plan / Preview | Rollback |
|------|---------|----------|----------------|----------|
| 是否产生真实副作用 | 否 | 可能 | 否 | 已产生后再撤回 |
| 核心价值 | 预览影响范围 | 验证行为 | 展示变更计划 | 降低损失 |
| 典型场景 | 批量删除、同步、部署、迁移 | 系统试运行、功能测试 | IaC 计划、配置预览 | 发布失败恢复 |

## 4. 判断标准

一个操作可以称为 Dry Run，通常需要同时满足：

1. 不写入、不删除、不发布、不发送、不提交。
2. 能展示真实执行时的主要影响对象。
3. 使用与真实执行相同或接近的参数解析逻辑。
4. 输出足够让人判断是否可以继续真实执行。

不满足第 1 条时，应称为测试运行、试运行或灰度执行，而不是 Dry Run。

## 5. 实践指南 (Practice Guide)

> [!IMPORTANT] 实践指南
>
> 1. 对批量删除、目录同步、数据库迁移、生产部署等高风险操作，先执行 Dry Run。
> 2. 检查输出中的对象数量、路径范围、过滤条件和权限错误。
> 3. Dry Run 通过后，再执行真实命令，并保留预演输出作为审查依据。
> 4. 不要把 Dry Run 当成成功保证；真实执行仍可能受时间差、权限变化、外部状态变化影响。

```bash
rsync -av --dry-run source/ target/
kubectl apply --dry-run=server -f app.yaml
```
