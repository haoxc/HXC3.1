## 一句话定义

上下文对齐是确保沟通参与者对讨论对象、当前进度、关键结论拥有一致理解的过程，是协作不跑偏的前置条件。

## 最小理解图

上下文对齐 = 三个问题的一致性检查：

| 对齐维度 | 回答的问题 | 不齐的表现 |
|----------|-----------|-----------|
| 概念对齐 | 你说的这个词是什么意思？ | 同一术语，不同理解 |
| 进度对齐 | 我们聊到哪了、结论是什么？ | 重复讨论或跳过结论 |
| 目标对齐 | 我们当前在解决什么问题？ | 各说各的，讨论发散 |

## 实现手段

| 维度 | 工具 | 时机 |
|------|------|------|
| 概念对齐 | [[口径]] | 讨论开始前 |
| 进度对齐 | [[Recap(概要重述)]] | 中断后继续前 |
| 目标对齐 | 议题明确 | 讨论偏离时拉回 |

## 判断标准

- 任意一个参与者被问「当前在讨论什么？关键结论是什么？核心术语的定义是什么？」能否给出与其他人一致的回答？
- 能 → 上下文已对齐。

## 典型例子

一个需求评审会，3 个人在讨论「接口对接方案」：
- A 说的「接口」是 REST API，B 理解的是 Java Interface → **概念没对齐**
- C 中途加入，不知道前面已否决了方案 1 → **进度没对齐**
- 讨论逐渐发散到数据库选型 → **目标没对齐**

→ 三个维度全不齐，会议结果不可靠。应先花 2 分钟对齐上下文再继续。

## 压缩记忆

> 上下文对齐 = 让三个人讨论时，不会有两个人在说不同的事。口径 + Recap 是其两个操作手柄。
