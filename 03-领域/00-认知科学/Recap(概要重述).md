## 一句话定义

Recap（Recapitulation）是对已讨论内容的关键点做简要重述，目的是让缺席者或中断后回来的人快速恢复上下文，无需重读全部历史。

## 最小理解图

| 场景 | 没有 recap | 有 recap |
|------|-----------|----------|
| 对话中断后继续 | 翻聊天记录找上下文 | 读 3 行摘要即可接上 |
| 新人加入讨论 | 「前面聊了什么？」 | 直接看 recap 对齐 |
| 长会议中场休息后继续 | 「刚才说到哪了？」 | 2 分钟 recap 全员复位 |

## 判断标准

- 缺席者能否仅凭 recap 理解当前讨论的**关键背景和待解决问题**？
- recap 是否只含已确认的结论和上下文，不含新信息？

两项都是「是」→ 合格的 recap。

## 如何写 recap

| 要素 | 规则 |
|------|------|
| 长度 | 3–5 句话，不超过一个屏幕 |
| 内容 | 只写**已达成的结论**和**当前待解决的事项** |
| 不含 | 过程辩论、已否决方案、新提议 |
| 位置 | 放在继续讨论的最开头 |
| 格式 | 「前面我们讨论了 X，达成了 Y，当前在解决 Z」 |

## 容易混淆

| 对比          | 区别                                                        |
| ----------- | --------------------------------------------------------- |
| recap vs 总结 | 总结是终局产出（覆盖全过程+结论）；recap 是中途同步（只覆盖中断前已讨论部分）                |
| recap vs 摘要 | 摘要是对一段内容的压缩复述（可对任何内容）；recap 特指对**讨论进程**的上下文重建             |
| recap vs 口径 | 口径对齐**概念定义**（你说的 X 是什么意思）；recap 对齐**上下文状态**（我们聊到哪了、结论是什么） |
|             |                                                           |

## 典型例子

- Claude Code 上下文压缩后，开头附一段会话摘要 → 这就是 recap
- 会议开到一半有人加入，主持人说「前面讨论了 A 方案和 B 方案，倾向 A」→ 口头 recap
- Git commit message 的 body 部分简述本次改动 → 对未来的自己做的 recap

## 压缩记忆

> Recap = 给后来者的一张「前面聊了什么」便签。不是总结全貌，而是补上上下文缺口。
