## 概述
- AI 与外部工具交互的「思维决策框架」
ReAct 的本质是 **“思维链（Chain of Thought）+ 工具交互” 的循环迭代**，核心解决两个问题：
1. 复杂任务无法单步完成（需拆分步骤，如 “数据分析→生成图表→解读结论”）；
2. 任务存在不确定性（需根据中间结果调整方向，如工具调用失败、数据不足时重试 / 补充信息）。
#### 与传统 “单步工具调用” 的区别：
- 传统方式：用户需求 → 模型直接调用工具 → 返回结果（无思考过程，失败即终止）；
- ReAct 方式：用户需求 → 模型思考 “第一步该做什么”→ 调用工具 → 观察结果 → 思考 “下一步该做什么”→ 迭代直到完成任务（有决策、有修正）。
#### 关键前提：
ReAct 依赖 **Prompt 引导**（让模型输出 “思维链 + 行动指令”）和 **工具支持**（模型可调用的外部服务，如搜索、数据库、API），常与 Function Calling 结合（用 Function Calling 标准化 “行动” 步骤）。
### 二、ReAct 核心循环：思考→行动→观察（3 步闭环）

ReAct 的核心是无限循环 “思考 - 行动 - 观察”，直到任务完成或触发终止条件。用 **“服务器故障排查”** 案例拆解每一步：
#### 任务背景：
用户需求 → “我的网站打不开了，提示‘502 Bad Gateway’，帮我排查原因并给出解决方案”（需调用 “服务器日志查询工具”“端口检测工具”“进程查询工具”）。

| 循环步骤           | 模型行为（ReAct 逻辑）                     | 具体示例                                                                                                                                                                                                                                                                                                                      |
| -------------- | ---------------------------------- | ------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------- |
| 1. 思考（Reason）  | 分析当前状态、明确下一步目标，解释 “为什么要这么做”        | 模型输出思维链：<br><br>“用户网站报 502 错误，可能是反向代理（Nginx）故障、后端服务未启动或端口占用。第一步应该先查询服务器 Nginx 日志，看是否有连接后端失败的记录，因此需要调用‘日志查询工具’，筛选近 10 分钟的 Nginx 错误日志。”                                                                                                                                                                                     |
| 2. 行动（Act）     | 调用工具（可结合 Function Calling 输出结构化指令） | 模型输出行动指令（结构化格式，方便系统解析）：<br><br>`json{"name":"query_log","parameters":{"log_type":"nginx","time_range":"10min","level":"error"}}`                                                                                                                                                                                          |
| 3. 观察（Observe） | 接收工具返回结果，判断是否满足下一步条件               | 工具返回：<br><br>“Nginx 日志显示：connect () failed (111: Connection refused) while connecting to upstream [127.0.0.1:8080]”（连接后端 8080 端口被拒绝）                                                                                                                                                                                      |
| 4. 循环迭代        | 基于观察结果，重复 “思考→行动→观察”               | 第二次思考：<br><br>“日志显示 8080 端口连接失败，可能是后端服务（如 Tomcat）未启动或端口被占用。下一步调用‘端口检测工具’，检查 8080 端口是否在监听。”<br><br>第二次行动：<br><br>`json{"name":"check_port","parameters":{"port":8080,"host":"127.0.0.1"}}`<br><br>第二次观察：<br><br>“8080 端口未监听，无进程占用。”<br><br>第三次思考：<br><br>“后端服务未启动，调用‘进程查询工具’检查 Tomcat 进程是否存在，若不存在则启动服务。”<br><br>... 直到问题解决 |

## ReAct 与 Function calling

| 维度       | ReAct（Reason + Act）                       | Function Calling（函数调用）              |
| -------- | ----------------------------------------- | ----------------------------------- |
| **核心定位** | 模型的 “思维决策框架”—— 指导模型如何**规划步骤、迭代修正**        | 模型与工具的 “通信协议”—— 定义模型如何**标准化调用工具**   |
| **本质**   | 一种 “思考 - 行动 - 观察” 的循环逻辑，模拟人类解决问题的过程       | 一种结构化输出格式，让模型将用户需求解析为 “工具名 + 参数”    |
| **核心目标** | 解决 “复杂多步骤任务”（如 “规划旅行 + 订酒店 + 查天气”），处理不确定性 | 解决 “工具调用的准确性”（如 “查北京明天天气”），避免自然语言歧义 |
| **依赖**   | 可独立使用（用自然语言调用工具），也可结合 Function Calling    | 依赖工具提供明确的 “函数定义”（名称、参数、格式），需模型适配    |
|          |                                           |                                     |
