# Agent 与 Skill 关系

> [!summary] 一句话
> **智能体(Agent)** 是执行任务的主体；**技能(Skill)** 是智能体在特定场景下可加载、可复用的专业能力包。

## 1. 概念界定

- **智能体(Agent)**：能够理解目标、规划步骤、调用工具、读写环境并产出结果的执行主体。
- **技能(Skill)**：面向某类任务封装的说明书和资源包，通常包含触发场景、工作流程、约束规则、参考资料、模板或脚本。
- **AGENTS.md(Agents Instruction File)**：定义 Agent 长期行为底座的指令文件，例如角色、语气、偏好、权限边界和输出格式。

## 2. 边界辨析

| 维度 | Agent | Skill |
| --- | --- | --- |
| 本质 | 执行任务的主体 | 可被调用的能力包 |
| 作用 | 理解目标、拆解任务、调用工具、完成交付 | 提供特定场景下的方法、流程、知识和约束 |
| 范围 | 全局、持续、覆盖所有任务 | 局部、按需、覆盖某类任务 |
| 主动性 | 主动判断、选择、执行 | 本身不执行，等待 Agent 识别并调用 |
| 生命周期 | 一次会话、一个线程、一个运行环境中的执行者 | 可长期沉淀、安装、复用、迁移 |
| 类比 | 人 / 工作者 / 项目执行者 | 方法书 / 工具箱 / 作业指导书 |

## 3. 在 Codex 中的三层关系

| 层级 | 对应对象 | 主要问题 | 例子 |
| --- | --- | --- | --- |
| 角色底座 | `AGENTS.md` | 这个 Agent 应该以什么风格和规则工作？ | 专业沉稳、先翻译提示词、使用 `---` 分隔 |
| 执行主体 | Codex Agent | 谁来理解任务并完成操作？ | 读文件、改 Markdown、调用 shell、更新 MOC |
| 专项能力 | Skill | 做某类任务时应该按什么方法做？ | `note-crafter` 写 Obsidian 笔记，`bx` 做概念辨析 |

简化理解：

```text
AGENTS.md 规定长期行为底色
        ↓
Agent 接收任务并决定怎么做
        ↓
Skill 在特定场景下提供专业方法
        ↓
工具与文件系统完成实际动作
```

## 4. 类比解释

### 类比一：医生与临床路径

Agent 像医生，负责判断病情、选择方案、执行诊疗。Skill 像某类疾病的临床路径，规定遇到特定问题时应该检查什么、先做什么、避免什么。

这个类比的限制是：Agent 不一定像人一样拥有长期稳定经验，很多能力依赖当前上下文、模型能力和可用工具；Skill 也不等于强制流程，Agent 仍需要根据现场情况调整。

### 类比二：项目经理与作业指导书

Agent 像项目经理或工程师，负责接收任务并推动完成。Skill 像作业指导书，告诉他在“创建笔记”“生成 PPT”“提交 Git”这类固定场景中应按什么规范执行。

这个类比的限制是：有些 Skill 会包含脚本和资源，不只是文字说明；它更接近“方法 + 资料 + 工具”的组合包。

## 5. 使用判断

需要强化 **Agent** 时，通常是在定义：

- 总体角色：它应像专家、编辑、工程师还是审计员？
- 沟通风格：简洁、解释充分、结论先行还是教学式？
- 全局边界：是否允许改文件、是否需要先确认、是否要保护用户已有修改？
- 输出格式：是否固定先翻译、列关键词、再输出正文？

需要创建或修改 **Skill** 时，通常是在固化：

- 高频任务流程：例如创建笔记、写调研报告、做 Git 提交检查。
- 领域方法论：例如概念辨析、课程审计、PRD 编写。
- 可复用模板：frontmatter、报告结构、表格字段、检查清单。
- 工具脚本和资源：例如渲染文档、生成图片、解析表格、调用专门 MCP。

## 6. 常见误区

- **误区：Skill 是另一个 Agent。** Skill 不是执行主体，它需要被 Agent 调用后才发挥作用。
- **误区：Agent 有了 Skill 就一定会自动做好。** Skill 提供方法，但仍依赖任务描述、上下文质量、工具权限和执行验证。
- **误区：所有偏好都应写成 Skill。** 全局稳定偏好适合写入 `AGENTS.md`；某类任务的专业流程才适合写成 Skill。
- **误区：Skill 越多越好。** Skill 应围绕高频、可复用、边界清晰的任务沉淀；过多低质量 Skill 会增加选择和维护成本。
