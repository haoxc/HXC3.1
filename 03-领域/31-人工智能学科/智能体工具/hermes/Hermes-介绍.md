# Hermes-介绍

## 一句话定义

Hermes 是一个面向长期运行的本地智能体编排层：它把模型、技能、记忆、任务调度、WebUI、CLI 和后台任务统一到一个可持续工作的 Agent 环境中。

## 它解决什么问题

Hermes 的核心价值不是“单次回答更强”，而是让 AI 任务可以跨会话、跨入口、跨时间持续运行。

典型问题：

- 今天交代的任务，明天还能接着执行。
- 同一个知识库、项目和工具栈，不需要每次重新解释。
- 后台巡检、定时任务、周期报告可以自动运行。
- 不同模型可以通过统一入口调用。
- WebUI、CLI、定时任务、消息入口共享同一套上下文和技能。

## 核心能力

| 能力       | 含义                  | 典型用途                     |
| -------- | ------------------- | ------------------------ |
| 多入口会话    | WebUI、CLI、后台任务等入口统一 | 在浏览器中启动，在终端中继续           |
| 持久记忆     | 跨会话保留项目、偏好、工具栈信息    | 不反复解释 Vault 结构           |
| Skill 系统 | 将专业流程沉淀成可复用能力       | 审计、写作、巡检、项目管理            |
| 后台任务     | 当前会话之外并行运行任务        | `/background` 只读检查       |
| 定时任务     | 按 schedule 自动触发     | 每日/每周 Vault 巡检           |
| 模型路由     | 通过配置选择不同模型          | OpenAI、Claude、DeepSeek 等 |
| 本地执行     | 在用户机器或服务器上运行        | 文件扫描、脚本、工具调用             |

## 和 Codex / Claude Code 的分工

| 工具 | 主要定位 | 更适合做什么 |
| --- | --- | --- |
| Hermes | 编排层、调度层、长期运行层 | 定时巡检、后台任务、任务分派、跨会话记忆 |
| Codex | 本地文件与知识库精细执行者 | Obsidian 笔记、规范整理、文件编辑、审计落地 |
| Claude Code | 代码仓库工程执行者 | 代码理解、实现、测试、工程重构 |
| LiteLLM | 模型路由层 | 统一管理模型供应商和模型别名 |
| Obsidian | 知识承载层 | 保存笔记、MOC、规范和审计报告 |

简化理解：

```text
Hermes = 调度员 / 值班员 / 入口路由器
Codex = Vault 笔记与本地文件执行者
Claude Code = 代码仓库执行者
LiteLLM = 模型路由层
Obsidian = 知识库承载层
```

## 适合使用 Hermes 的场景

- 需要后台或定时执行的任务。
- 需要长期记住项目上下文的任务。
- 需要在 WebUI 和 CLI 之间切换的任务。
- 需要统一管理多模型、多技能、多工作区的任务。
- 需要把巡检、提醒、日报、周报做成自动化流程。

## 不必使用 Hermes 的场景

- 只问一次、一次性回答即可的问题。
- 当前文件内的精细编辑，直接用 Codex 更顺手。
- 单个代码仓库的实现和测试，Claude Code 或 Codex 更直接。
- 需要人工反复判断的长文写作，不适合完全后台化。

## 常用入口

| 入口 | 用法 | 说明 |
| --- | --- | --- |
| WebUI | `http://localhost:8787` | 浏览器界面，适合会话和任务管理 |
| CLI | `hermes` | 终端入口，适合命令行工作流 |
| 后台任务 | `/background <prompt>` | 并行运行，不阻塞当前会话 |
| 旁路问题 | `/btw <question>` | 临时提问，不污染主历史 |
| 定时任务 | `Tasks / Scheduled jobs` | 周期执行和离线运行 |

## 风险边界

Hermes 越适合长期运行，越需要明确权限边界。

默认建议：

- 后台任务先只读。
- 批量移动、删除、重命名前必须人工确认。
- 大范围写入前先生成计划和 diff。
- 自动任务必须留下日志。
- 高风险操作交给 Codex 或 Claude Code 精细执行。
