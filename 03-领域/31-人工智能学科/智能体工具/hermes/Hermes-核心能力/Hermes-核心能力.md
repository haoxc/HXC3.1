## 一句话结论

Hermes 的价值在于长期运行、`跨入口触发`、`任务分派`和模型路由；主要风险也来自这些能力：一旦边界不清，它可能把“自动化便利”放大成“不可控执行”。

## 背景

前一轮对 Hermes 核心能力的判断是：

```text
Hermes = 调度员 / 值班员 / 入口路由器
Codex = Vault 笔记与本地文件精细执行者
Claude Code = 代码仓库与工程任务执行者
LiteLLM = 模型路由层
Obsidian = 知识库承载层
```

这套分工成立的前提是：Hermes 不被误用成“什么都自动执行的万能代理”，而是承担清晰的编排、巡检、提醒和低风险动作。

## 核心关注点

| 关注点      | 风险                                       | 应对策略                                    |
| -------- | ---------------------------------------- | --------------------------------------- |
| 自动触发     | Cron、Webhook、消息入口可能在用户不在场时触发任务           | 默认只读；写入任务必须有白名单和日志                      |
| 权限边界     | Agent 获得过多文件、命令或密钥访问权                    | 按任务最小授权；敏感目录、删除命令、密钥文件禁止自动执行            |
| 工具分工     | Hermes 直接做精细 patch，容易越界                  | Hermes 负责发现和分派，精细修改交给 Codex/Claude Code |
| 模型路由     | 不同模型能力、价格、上下文和工具调用兼容性不同                  | 通过 LiteLLM 固定模型别名，按任务类型选择模型             |
| Skill 漂移 | Hermes/Codex/Claude Code 三套 skill 内容逐渐分叉 | 核心规范一份，运行时适配分开维护                        |
| 可观测性     | 自动任务失败后不容易发现                             | 每次任务输出日志、状态、失败原因和下次动作                   |
| 结果可信度    | Agent 巡检可能误判、漏判或过度建议                     | 高风险动作前增加人工确认；报告保留证据路径                   |
| 成本控制     | 长期任务可能消耗模型额度                             | 定时任务分级：只读扫描优先本地脚本，必要时再调用模型              |

## 不应自动化的动作

以下动作不建议由 Hermes 在无人确认时自动执行：

- 删除文件、移动大量目录、清理附件。
- 修改 `.obsidian/` 配置。
- 批量重命名笔记或改写双链。
- 修改 Git 历史、自动提交并推送。
- 修改生产环境配置、密钥、数据库。
- 对大范围 Markdown 做风格统一或语义改写。

这些动作可以由 Hermes 发现问题并生成建议，但执行前应交给用户确认，再由 Codex 或 Claude Code 精细落地。

## 推荐安全分级

| 等级 | 允许 Hermes 自动做什么 | 示例 |
| --- | --- | --- |
| L0 只读观察 | 读取目录、统计、生成报告 | 检查 MOC 漏挂、frontmatter 缺失、重复主题 |
| L1 低风险写入 | 写入固定 inbox、日志、日报 | 写入 daily note capture、生成巡检报告 |
| L2 受限修复 | 只修复白名单文件和明确模板项 | 给新笔记补 frontmatter、给 SVG 补说明页 |
| L3 人工确认执行 | 生成计划，等待确认后执行 | 批量移动、重命名、修改索引 |
| L4 禁止自动执行 | 只允许报告，不允许执行 | 删除、密钥、生产配置、Git 强制操作 |

## Hermes 适合承担的任务边界

### 适合

- 定时巡检。
- 捕获输入。
- 生成任务清单。
- 判断任务该交给谁。
- 调用 LiteLLM 做模型路由。
- 维护自动化运行日志。
- 对低风险模板化内容做补全。

### 不适合

- 当前工作区内高精度编辑。
- 大规模文件重构。
- 长文逐段打磨。
- 需要持续对话判断的写作任务。
- 需要强人工把关的生产环境操作。

## 落地前验证清单

在让 Hermes 接管一个自动化任务前，至少确认：

- 任务是否可以先只读运行一周？
- 失败时是否会留下可追踪日志？
- 写入文件是否在白名单目录内？
- 是否有最大修改数量限制？
- 是否禁止删除与覆盖？
- 是否区分报告、建议、执行三个阶段？
- 是否明确由 Codex/Claude Code 接手精细修改？
- 是否能通过 Git diff 看清全部变更？
- 是否有模型调用成本上限？
- 是否有关闭开关？

## 与 Codex / Claude Code 的分工

| 任务          | Hermes  | Codex | Claude Code |
| ----------- | ------- | ----- | ----------- |
| 每周 Vault 巡检 | 主控      | 可执行修复 | 不优先         |
| 新建普通笔记      | 可捕获、可草拟 | 主执行   | 不优先         |
| MOC 精修      | 发现问题    | 主执行   | 不优先         |
| 代码仓库任务      | 分派      | 可辅助   | 主执行         |
| Git 提交推送    | 提醒/检查   | 主执行   | 可执行         |
| 模型配置巡检      | 主控      | 可编辑文件 | 可辅助         |

## 最小落地策略

1. 第一阶段只做 L0 只读观察：每周生成 Hermes/Vault 巡检报告。
2. 第二阶段开放 L1 写入：允许写入固定日志或 inbox，不改原笔记。
3. 第三阶段开放 L2 受限修复：只补明确模板字段，不做语义重写。
4. L3 以上动作一律先生成计划，等待用户确认后再执行。

## 压缩记忆

> Hermes 的关键不是“更会执行”，而是“更会长期触发和分派”；越自动，越要把权限、日志、白名单和人工确认设计在前面。
